# Board 03 — CONSTRUCTOR / AUTHORING (the no-code visual builder)

> Senior platform-architect deep-analysis. Scope: `platform/apps/panel` (the whole Constructor) + the engine coverage-fitness gate and registries it consumes. Strategy: read the design intent (`docs/architecture/subsystems/15-constructor.md`, `future/03-constructor`, `future/04-dep-graph`, `future/05-async-options`, `24-options-source`, `30-plugin-taxonomy`) then VERIFY against code. Analysis only — no product code changed.
>
> Headline: the Constructor is **architecturally first-class and substantially ahead of the documented design**. The original `15-constructor.md` spec (iframe canvas, JSON-Schema forms, raw-JSON fallback for un-schema'd types) has been *superseded* by a stronger build: an in-process WYSIWYG canvas rendering the real engine, a schema-driven generic Inspector, and an enforced **coverage gate** that mechanically proves "every renderer capability has an authoring surface." The remaining work is mostly **polish + two latent i18n architecture holes**, not missing capability.

### [CON-01] Coverage Gate — "every engine capability is authorable, or an explicit shrinking TODO"
- **Status**: ✅DONE (the platform's strongest single asset)
- **Evidence**: `platform/apps/panel/src/features/data-layer/coverage.fitness.test.ts:1-261`; SSOT enumerators in `platform/packages/core/src/config/discriminant-manifest.ts` (`DATASPEC_DISCRIMINANTS`/`PARAMDEF_TYPES`/`VISIBILITY_OPS`), `step-registry.ts:55` (`listTransformOps()`), `perspective-scope-registry.ts:50` (`listPerspectiveScopeKeys()`).
- **What & why**: A fitness test enumerates *five* capability axes **from the engine SSOT** (not a hand-list) and asserts each member is either surfaced by an authoring control OR sits in `COVERAGE_TODO`, the roadmap-keyed allowlist. A new un-surfaced op/type **fails the build**. `COVERAGE_TODO` is currently down to permanent escapes only (`joinByField`, DataSpec `custom`) — params/visibility/dataSpec allowlists are *empty*, meaning full coverage. A 6th test catches stale allowlist entries (drift in the other direction).
- **Critical analysis** (skeptical): This is the right pattern and rare in the wild — but it proves *existence of a surface*, not *quality or completeness of that surface*. `paramDefAuthorable(type)` is `getParamSchema(type) != null`: a schema with one trivial field passes. The gate cannot detect a surface that is present-but-unusable (e.g. an enum-ref with an unreachable source). It also gates by *discriminant*, not by *field*: a DataSpec type whose editor omits half its fields still passes. Coverage is structural, not semantic — a known and acceptable limit, but it must not be mistaken for "the UX is good."
- **Reference platforms**: No mainstream builder (Builder.io, Retool, Grafana) ships a mechanical "renderer-capability ⊆ authoring-surface" gate — they discover gaps via support tickets. **We beat all of them here**: this is a genuine fitness-function-driven evolutionary-architecture move (skill §5/§12). Grafana's closest analogue is manual editor registration with no closure proof.
- **Foresight** (1–2yr; multi-tenant): As tenants register custom node types/plugins, the gate must run *per registered plugin set*, not just the built-in barrel. Today it imports `@statdash/engine` module-init; a tenant plugin that adds a transform op but no schema would only fail if its registration is in the test's import graph. Multi-tenant authoring needs the gate parameterized over "the active capability manifest" (the `describeApp()` payload, `constructor.ts:140`).
- **Plan**: (1) Add a *semantic depth* companion fitness: for each DataSpec discriminant, assert its editor renders ≥ the required fields of that type (cross-check against the engine PropSchema `required` set). (2) Wire the gate to consume `describeApp()`/`CONTRACT_VERSION` so it travels with the published contract. Files: `coverage.fitness.test.ts`, new `coverageDepth.fitness.test.ts`, `packages/react/src/engine/constructor.ts`. Fitness-fn: "every required field of every union member has a control." Effort: M. Risk: two-way. Class **G**. Priority: **P1**.
- **Raises-the-bar**: Promote the gate from "a surface exists" → "a *complete, reachable* surface exists" — the difference between Form.io claiming a field type and JSON-Forms proving the form validates.

### [CON-02] Schema-driven generic Inspector + FieldControlRegistry (C1)
- **Status**: ✅DONE
- **Evidence**: `platform/apps/panel/src/inspector/Inspector.tsx:1-177`, `FieldControlRegistry.ts:1-100`, `fieldControl.types.ts`, `schemaSource.ts`, `controls/primitives.tsx`, `controls/EnumRefField.tsx`, `controls/LocaleField.tsx`; engine seam `platform/packages/core/src/config/prop-schema.ts:92-132`.
- **What & why**: The whole property panel renders **generically** from `schemaSource.getSchema(node)` → no per-type form. Field dispatch is a Strategy+Registry: `PropFieldType → FieldControl`, with a documented precedence (localized variant → enum-ref → static-select → by-type → raw-JSON fallback). A new node type = a new schema (zero Inspector change); a new field type = one `register()` (zero Inspector change). `SchemaSource` (Dependency Inversion) lets the SAME Inspector serve nodes, chrome, page, perspectives, params, visibility leaves — six surfaces, one renderer. This is the JSON-Forms/RJSF/Retool canon, done with a typed in-house DSL (`PropSchema`) rather than raw JSON Schema (richer: `enum-ref` data-driven sources, `coverage:'localized'`, `showWhen`, groups).
- **Critical analysis**: Strong. Two seams to watch: (a) the registry is keyed by *runtime string* to accept engine types the panel's TS doesn't know yet — pragmatic, but it means a typo'd type silently falls to JsonControl rather than erroring (Postel-liberal; acceptable but should be observable). (b) `DataSpec`/`ChartDef`/`object`/`array` default to `JsonControl` in the base registry (`FieldControlRegistry.ts:97-99`) and are *overridden* by app registrations (DataSpecEditor etc.) — correct, but the "raw JSON for a power-type" fallback is one missed registration away from leaking raw JSON into a non-programmer's face. There is no fitness test asserting "no shipped node schema resolves a field to JsonControl."
- **Reference platforms**: Retool/Appsmith property inspector, Grafana's `PanelOptionsEditorRegistry`, Builder.io `inputs`. **We beat them** on the open `SchemaSource` port (one Inspector across 6 element kinds) and on `enum-ref` *runtime catalog* sources (`prop-schema.ts:61-70`: cube.measures/dimensions/members, dataSpecs, tokens, pages, filterParams, perspectives) — Builder.io's inputs are static lists; ours bind live to the connected cube.
- **Foresight**: Multi-tenant theming/branding of the Inspector itself; conditional field *groups* (collapsed accordions exist; cross-field show/hide is per-field `showWhen` only — no group-level conditions yet). 1–2yr: a `ConstructorEditorProps<T>` "custom editor slot" (designed in `future/03-constructor/editor-slot-impl.md`, not yet implemented) for power fields that outgrow generic controls.
- **Plan**: (1) Add fitness: "every registered node PropSchema field resolves to a non-fallback control (or is explicitly allowlisted as raw-JSON)." (2) Implement the `editor?` slot from the design doc for `DataSpec` rich editing parity. Files: new `inspector/controlCoverage.fitness.test.ts`, `FieldControlRegistry.ts`, `fieldControl.types.ts`. Fitness-fn above. Effort: S/M. Risk: two-way. Class **G**. Priority: **P2**.
- **Raises-the-bar**: A "no raw JSON in the non-programmer path" guarantee, enforced — most builders quietly drop to a code editor; we'd prove we don't.

### [CON-03] Capability-gated palette (C3) — dataset-aware discovery
- **Status**: ✅DONE
- **Evidence**: `platform/apps/panel/src/discovery/capabilityGate.ts:1-71`, `capabilityGate.test.ts`, `discovery/suggestPanels.ts`, `discovery/cubeProfile.store.ts`, `discovery/useActiveProfile.ts`; palette `canvas/paletteEntries.ts`, completeness pinned by `canvas/paletteCompleteness.test.ts`.
- **What & why**: The palette is derived 100% from `nodeRegistry.list()` (OCP: a registered type appears with zero palette code — pinned by `paletteCompleteness.test.ts`). On top, `gatePaletteEntries` removes data-bound panels the active cube profile *cannot* support (no measure → no chart; geo panel needs a geo-role dimension). Gating is **conservative/Postel** (only removes what it can PROVE unsupported) and **fail-open** (no profile → full palette) — graceful degradation. Data-binding capability is read from declared `caps` + concept roles, never a hardcoded panel-type list (Law 1).
- **Critical analysis**: `needsGeo` is heuristic (`entry.type === 'map' || type.includes('geo') || caps.includes('map')`, line 40) — a string-contains check is a smell in an otherwise capability-clean module; a future `geo-choropleth` type works by luck, a `regional-bar` matches `geo` by accident. Should be a declared `requiresRole: 'geo'` cap. Also: gating is binary keep/drop; best-in-class builders *dim + tooltip* ("needs a measure") rather than hide, so the author learns *why* a capability is missing (discoverability > silent absence).
- **Reference platforms**: Tableau "Show Me" greys-out unavailable viz with a reason; Power BI similar. **They beat us** on *explaining* the gate (we hide silently); **we beat them** on the gate being a pure, tested, capability-declared function rather than hardcoded viz rules.
- **Foresight**: Multi-cube pages (a page bound to 2 datasets) — the gate is single-`ActiveProfile` today. 1–2yr: per-node profile context as blend/multistore land (`COVERAGE_TODO.perspectiveScope` already names `store`/`blend` doors).
- **Plan**: (1) Replace `needsGeo` string-match with a declared `requiresRole` capability on the palette entry (read from node meta). (2) Switch hide → *disable + reason tooltip* ("ამ პანელს სჭირდება საზომი"). Files: `capabilityGate.ts`, `paletteEntries.ts`, `NodePalette.tsx`, plugin `meta.ts` files. Fitness-fn: "no entry is gated by a string-substring of its type." Effort: M. Risk: two-way. Class **G**. Priority: **P2**.
- **Raises-the-bar**: Capability gating that *teaches* (Tableau-grade affordance) on a fully declarative substrate.

### [CON-04] Round-trip fidelity — flat store ⇄ tree config, byte-identical
- **Status**: ✅DONE
- **Evidence**: `platform/apps/panel/src/canvas/canvasPageAdapter.ts` + `.test.ts`, `save/saveGuard.ts` (`stableStringify`), `save/authorRender.e2e.test.tsx:136-152` (lossless round-trip assertion), `command/insertByteIdentity.fitness.test.ts`, `store/hierarchyPatches.test.ts`, `store/nestedPropWrite.test.ts`.
- **What & why**: The store holds a *flat* node model; `toNodePageConfig`/`fromNodePageConfig` project to/from the engine's nested tree. The e2e fitness asserts `stableStringify(fromNodePageConfig(toNodePageConfig(page))) === stableStringify(page)` AND that re-projection is identical — the lossless visual↔JSON invariant (skill §12 SSOT). A separate fitness asserts an insert is **byte-identical regardless of surface** (DnD vs Cmd-K vs slash vs Outline all go through one `makeNode`/`insertNode` path — `PageStep.tsx:76-85`). The save-guard's four checks (validate + round-trip + per-node-valid + locale-complete) are the publish gate.
- **Critical analysis**: This is the correct invariant and well-tested. Risk surface: round-trip identity depends on `setVisibleWhen` carefully deleting emptied `view`/`visibleWhen` keys to avoid `undefined` leakage (`PageStep.tsx:109-125`) — a hand-maintained cleanliness rule that a future field could forget, silently breaking byte-identity. This argues for a *generic* "prune empty/undefined branches" pass at serialization rather than per-field deletes. No fuzz/property-based round-trip test (only fixed fixtures) — a property test over arbitrary valid trees would harden it.
- **Reference platforms**: Builder.io's content JSON is the canonical SSOT with lossless round-trip; Webflow likewise. **Par with the leaders**; **we beat** most form builders (Form.io/RJSF) which don't guarantee byte-identity across the editor boundary.
- **Foresight**: Schema versioning — when `schemaVersion` bumps, the round-trip must hold *across* a migration (load v1 → migrate → save v2 → never silently corrupt v1 authors). Expand-contract discipline needed (skill §12 versioning).
- **Plan**: (1) Add property-based round-trip (fast-check) over generated valid trees. (2) Replace per-field empty-key deletes with one generic serialization prune + a fitness "no serialized config contains an `undefined`/empty `view`." Files: `canvasPageAdapter.ts`, `saveGuard.ts`, new `roundTrip.property.test.ts`. Effort: M. Risk: two-way. Class **G**. Priority: **P2**.
- **Raises-the-bar**: Property-proven losslessness + a generic cleanliness pass = round-trip that can't rot field-by-field.

### [CON-05] Live WYSIWYG preview with real engine + opt-in live store injection (G3)
- **Status**: ✅DONE (and *better than the documented iframe design*)
- **Evidence**: `platform/apps/panel/src/canvas/CanvasView.tsx`, `CanvasLivePreview.test.tsx`, `useLivePreviewStores.ts:1-138`, `livePreview.ts`, `useDebouncedLivePage.ts`, `PageStep.tsx:182-202` (canvas mounts the real `NodePageRenderer`).
- **What & why**: `15-constructor.md` specified an **iframe of the actual app** (`?draft=true`). The build *surpassed* this: the canvas mounts the **real `@statdash/react` `NodePageRenderer` in-process** (same registry, same shells, same CSS — one SSOT, no iframe latency/handshake). `useLivePreviewStores` is the single seam choosing `structural` (empty `staticStore`, byte-identical zero-fetch default) vs `live` (real cube data via the SHARED `buildStoreManifest` the geostat runner uses — Law 3). Live mode is **fail-soft**: any failure (no cube-bound source, profile not ready, build throws) degrades to structural + a non-blocking `unavailable` badge, with a monotonic token guarding stale async builds. Debounced live page keeps typing responsive.
- **Critical analysis**: In-process render is the right call (Builder.io uses iframe mainly for style isolation; our engine + design tokens already scope cleanly, so the iframe tax isn't worth paying) — but it means a *runtime error in a node renderer can crash the editor frame* unless every node sits under an error boundary in the canvas path. Verify `NodeErrorBoundary` wraps canvas nodes as it does the runner (the doc promises it; the canvas must inherit it). The iframe also gave free *responsive-viewport* preview (mobile/tablet widths) — losing the iframe, do we still offer device-width preview? Not evident. Builder.io/Webflow treat responsive preview as table-stakes.
- **Reference platforms**: Grafana (live panel render), Retool (live component render), Builder.io (iframe). **We beat Builder.io** on fidelity/latency (no iframe handshake, identical module graph) and match Grafana/Retool; **they beat us** on built-in responsive device-frame preview.
- **Foresight**: Multi-tenant live preview must inject the *tenant's* store credentials/scope; the `buildStoreManifest` seam already supports this. 1–2yr: collaborative/multi-cursor preview (Figma-class) once the store is server-synced.
- **Plan**: (1) Confirm/add `NodeErrorBoundary` per node in `CanvasView` (a single bad node must not white-screen the editor). (2) Add device-width preview frames (mobile/tablet/desktop) to `CanvasToolbar`. Files: `CanvasView.tsx`, `CanvasToolbar.tsx`. Fitness: "a throwing node renderer in the canvas shows an inline error, not an unmounted editor." Effort: M. Risk: two-way. Class **G**. Priority: **P1**.
- **Raises-the-bar**: Iframe-free fidelity *plus* responsive frames + per-node resilience = Webflow-grade preview without the iframe tax.

### [CON-15] 🆕GAP — PerspectiveOption.label localization architecture (resolve-at-render vs freeze-at-parse)
- **Status**: 🆕GAP (architectural seam, flagged)
- **Evidence**: Authoring/wire `PerspectiveDef.label: ContractLocaleString` (`packages/contracts/src/perspective-axis.ts:62`) and core `LocaleString`. But the *render-time* `PerspectiveOption.label: string` (single-locale) — `packages/core/src/perspective/types.ts:21`. `PerspectiveBarShell.tsx:30` renders `{def.label}` from the parsed `available: PerspectiveOption[]`.
- **What & why**: `PerspectiveDef.label` is correctly multi-locale through authoring → wire → core. The question is *where* it collapses to a single string for the toggle. `PerspectiveOption.label` being `string` means the LocaleString is resolved **during axis parse** (`perspectiveOptions(axis)`), which freezes the toggle label to the locale active at parse time. If the page's active locale can change without re-parsing the axis (client locale switch), the perspective toggle would show a stale-locale label while everything else updates — the classic resolve-too-early bug. The rest of the engine resolves LocaleString **at render** with the active locale (the Inspector's `resolveLabel` pattern).
- **Critical analysis**: This is a small but real SSOT/timing inconsistency: one i18n carrier (perspective toggle) resolves at parse, all others at render. The fix is to carry `label: LocaleString` on `PerspectiveOption` (or have the shell resolve from the `PerspectiveDef` directly with the active locale) so resolution happens once, at render, everywhere. It's the same class as CON-14 (partial adoption of the i18n standard). Low blast radius today (server-rendered, single locale per request) but a latent defect the moment client-side locale switching ships — and the architecture should be uniform regardless.
- **Reference platforms**: Grafana variable labels, Power BI bookmark names — both single-locale (not a bar we need to clear). **We'd beat them** by making the perspective toggle locale-reactive like the rest of the engine.
- **Foresight**: Client-side locale toggle (a single page serving ka/en without reload) makes this load-bearing. Multi-tenant locale sets beyond ka/en.
- **Plan**: Change `PerspectiveOption.label` → `LocaleString` (or resolve in `PerspectiveBarShell` via the active-locale resolver, dropping the pre-resolution in `perspectiveOptions`). Verify one resolution site (render-time) for all i18n carriers — a "resolve LocaleString at render, never at parse" fitness/lint. Files: `packages/core/src/perspective/types.ts`, `registry.ts`, `PerspectiveBarShell.tsx`. Effort: S. Risk: two-way (small type change + one render site). Class **G**. Priority: **P2**.
- **Raises-the-bar**: One i18n resolution discipline (render-time) across the entire engine — no carrier resolves early.

## NET-NEW authoring innovation no builder ships (ambitious + YAGNI-honest)

**"Capability-Diff Authoring Trace" — a live, per-page proof that the page uses only what the connected data + registered capabilities can support, surfaced *as you build*.**

The platform already owns the two halves no competitor has together: (1) the **coverage gate** (engine-capability ⊆ authoring-surface, mechanically enumerated from SSOT) and (2) the **capability-gated palette** (dataset profile → supported panels). Fuse them into an authoring-time **trace panel**: for the page being built, show the *exact set* of engine capabilities it depends on (this DataSpec needs measure `B1G`; this perspective needs the `metric` scope key; this visibility leaf binds param `region`), each linked back to its SSOT registration and its data prerequisite, with a live red/green on whether the connected cube + registered plugin set satisfies it.

Why no one has it: Builder.io/Retool/Tableau don't have a closed capability SSOT to diff against — their renderers are open-ended, so "does this page use only supported capabilities?" is unanswerable. We *can* answer it because every capability is registered and the coverage gate already enumerates them.
