## MEDIUM

### MED-1 — The `effects` / `applyEffects` subsystem is orphaned dead code (Law 6, Lehman, item #8)
**Where:** `core/config/filter-validator.ts:119` (`applyEffects`, exported via `config/index.ts:33` + `core/index.ts:211`); `useFilterState.ts:213` (`schema?.effects ?? NO_EFFECTS`); `FiltersContext.tsx:17` (`FiltersCtx.effects`); `types/context.ts:75` (`RenderContext.effects`); `targets/html.tsx:173,276,307`; `SiteRenderer.tsx:118-124,284`.
**Root cause:** P6 deleted the only *caller* of `applyEffects` (the System-A mode-clearing `modeSet` path) but left the entire generic mechanism wired. Verification: `applyEffects(` has **zero** non-test callers (grep), and the sole consumer of the threaded `effects` — `FilterBarShell.tsx:12` — reads only `bars`, never `effects`. Provisioning declares **zero** `effects`.
**Why it matters:** This is exactly the "no surviving legacy / no dead code" the batch claims. Worse than inert: a config author who declares `filterSchema.effects: [...]` gets a **silent no-op** (Principle of Least Astonishment / fail-silent). The type, `applyEffects`, `NO_EFFECTS`, and the 3-layer context threading all rot in place.
**Canonical fix (not a patch):** Decide the mechanism's fate explicitly. Either (a) **delete it wholesale** — `Effect`, `applyEffects`, `schema.effects`, `FiltersCtx.effects`, `RenderContext.effects`, the `useFilterState` return, the html threading (YAGNI: no caller, no config) — or (b) if a generic declarative filter-effect capability is genuinely wanted post-System-A, **re-wire a caller** (a `filter:set` effect-application step) and add a fitness test asserting a declared effect actually fires. Do not leave it caller-less. The stale "Step 3.2d / Pass 2" follow-up comment (`useFilterState.ts:149-156`) referencing this dead path should go with it.

### MED-2 — `ManifestMode` / `SiteManifestContract.modes` survive: the "ALL System A grep-clean" claim is overstated (item #7, flagged for judgment)
**Where:** `packages/contracts/src/manifest.ts:24-27,70` (`ManifestMode`, `modes: ManifestMode[]`); `apps/api/src/routes/bootstrap/index.ts:35,94,269` (`DEFAULT_MODES`, `pick(SITE_KEY.modes,…)`); consumer `apps/geostat/src/app/App.tsx:40` (`boot.manifest.modes.forEach(m => perspectiveRegistry.register(m))`).
**Root cause:** The P6 grep-zero acceptance was scoped to `packages/**` + `apps/api/provisioning/**` — which **excludes** the site-scoped vocabulary in `contracts` + `apps/api/src/routes/bootstrap` + the `apps/geostat` boot. The architect's own full-stack plan called for repurposing this to `perspectiveKinds` / `ManifestPerspectiveKind` with a `site_config.modes → perspective_kinds` data migration; that rename was **deferred and not done**. So the System-A *name* `modes`/`ManifestMode` still ships on a live contract surface.
**Why it matters:** Not a correctness bug and not dead (the `manifest.modes → perspectiveRegistry → Constructor palette` chain is live, though `manifest.modes` is currently `[]` — `site-manifest.test.ts:40`). But it contradicts the P6/ADR framing of "zero legacy, grep-clean ALL System A," and it keeps the privileged-`mode` vocabulary the whole reframe set out to eliminate alive in the contract/API tier. This is the kind of survivor the mandate asked be flagged for judgment.
**Canonical fix:** Complete the planned strangler step — Postel-alias `ManifestMode → ManifestPerspectiveKind` and `SiteManifestContract.modes → perspectiveKinds` (additive, keep `modes` reading until the runner migrates), with the `site_config.modes → perspective_kinds` data migration — OR, if site-scoped kinds are unneeded now that the page axis owns presentation (decision B), delete the field+plumbing. Either way, retire the name; don't leave it as a System-A vocabulary island outside the grep scope.

## LOW

- **LOW-1 — Inert hardcoded `'mode'` fallback literals.** `SiteRenderer.tsx:99` (`perspectiveKeyPre … ?? 'mode'`) and `FiltersContext.tsx:20` (`EMPTY.perspectiveKey: 'mode'`). Benign (only used as a Record key when no axis is declared; never branched on), but they are privileged-dimension literals the reframe meant to erase. Also `perspective-state.ts:42` `LEGACY_MODE_PARAM='mode'` is load-bearing for the param-less `perspective-is` convention. Promote to a named `CONVENTIONAL_PERSPECTIVE_PARAM` constant or derive, so the convention has one home.
- **LOW-2 — `kpiVisible` evaluates `when` against `ctx.dims`, not `ctx.filterParams`.** `kpi.ts:209` passes `ctx.dims` as `evalVisibility`'s `fr` arg, whereas node visibility (`renderNode`) passes `ctx.filterParams`. Moot today (every kpi `when` is a param-less `perspective-is`, which ignores `fr`), but a future `eq`/`isset` kpi `when` would behave differently from the same op on a node — a latent least-astonishment trap. warm===render still holds (both kpi sites use the same `ctx.dims`). Align the `fr` source with `renderNode`'s, or document the divergence.
- **LOW-3 — `perspective-bar` carries the legacy partial ARIA-tabs pattern (a11y).** `PerspectiveBarShell.tsx:20-32` is byte-identical to the deleted `mode-bar` (`role="tablist"`/`role="tab"`/`aria-selected`) — so this is **preserved parity, not a regression** (FF-PERSPECTIVE-BAR-EQUIV). But the pattern is incomplete vs WAI-ARIA APG: no roving `tabindex` / arrow-key navigation, no `aria-controls`→tabpanel. Declaring `role="tab"` while implementing only part of the pattern is the "partial standard" Law 4 forbids. Pre-existing debt worth a dedicated a11y door: either complete the tabs pattern or drop to `role="group"` + `aria-pressed` toggle buttons (the honest semantics for state-switching buttons).
- **LOW-4 — Single-locale authoring labels (Law 4, authoring surface).** `setupCanvasRegistry.ts:45-46` registers ka-only labels (`'წლიური'`/`'დინამიკა'`); `perspective-bar/default/meta.ts:8` palette label is `{ka:'Perspective Tab Bar', en:'Perspective Tab Bar'}` (ka untranslated). End-user strings (PerspectiveDef labels, aria-label) ARE properly bilingual; this is authoring-UI-only, hence LOW — but the ka canvas labels should be `{ka,en}` and the palette ka should be a real Georgian string.
- **LOW-5 — Legacy fixtures/comments referencing `timeMode`/`modeOrder`.** `apps/api/src/provisioning/__fixtures__/legacy-filter-schemas.ts` + `perspective-migration-equiv.fitness.test.ts` intentionally carry `timeMode`/`modeOrder` to test the migration-equiv path. **Legitimate** (Postel/migration test inputs) — noted only to confirm they were reviewed and are not stray survivors.

## Verdict

This batch is best-in-class on the load-bearing axes: Law 1/2/3/9, the warm===render kpi SSOT, the permalink-from-registry guarantee, and the Constructor authoring round-trip are all genuinely clean and fitness-locked — the green is real, not cosmetic, and the hardest invariants (kpi-strip cold-throw, byte-identical toggle/nav, default-elision) are structurally enforced by single shared functions rather than by parallel code that could drift. **No HIGH items block shipping.** The two MEDIUMs are incomplete-retirement hygiene: an orphaned `effects`/`applyEffects` subsystem (dead + a silent-no-op footgun) and a surviving `ManifestMode`/`site_config.modes` vocabulary island in the contracts/API tier that makes the "ALL System A grep-clean" claim true only within its declared grep scope. Both are contained and alias-able, not correctness risks — but each should be closed (delete-or-rewire / complete-the-rename) before the "zero legacy" claim is recorded as fact, lest the next reader trust a grep scope that quietly excludes two live System-A surfaces.
