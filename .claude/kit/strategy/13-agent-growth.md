# 13 — Agent Growth: the team gets sharper with every path it walks

> **The owner's mandate:** agents must DEVELOP — become better versions of themselves, learning from
> projects, from the road they travel, from the reference platforms, from information sources, from
> observation. Always better, always sharper. This file is the mechanism, not a wish.

## The growth ladder (how a lesson climbs from experience to law)

```
1. EXPERIENCE      → the run itself (a defect met, a technique found, a benchmark read, a correction taken)
2. MEMORY          → the agent distills it into its OWN memory (feedback/project/reference — distillate, not log)
3. RECURRENCE      → the lead sees the same lesson twice (growth notes, ledger)
4. IDENTITY / DOCTRINE → a def-delta (this agent's craft sharpened) or a kit rule (the WHOLE team learns)
5. MACHINE         → if checkable, it becomes a hook / fitness gate (GUARDRAILS Ring 1–2) — learning that
                     matters stops depending on anyone remembering it
```
A lesson that stops at step 1 is lost; one that reaches step 5 can never be unlearned. The lead's job
is to keep lessons CLIMBING.

## The four learning sources (all mandatory reflexes, none a ritual)

1. **The road walked (experience).** Every significant run ends with a **GROWTH NOTE** in the return
   packet (shape below): what I'd do better next time · a failure mode I met · a technique that paid.
   Corrections AND validated successes both count — a team that records only its mistakes grows timid;
   one that records only wins grows blind.
2. **Reference platforms & information sources (research).** "Research when unsure" is already every
   def's reflex. The addition: research is HARVESTED — the distilled state-of-the-art finding goes into
   the agent's memory AND the return packet, so the team pays the research cost ONCE (the packet
   pipeline carries it to whoever needs it next). A benchmark insight left in a dead transcript is a
   team loss.
3. **Observation (the hunting-dog duty).** Everything surfaced beyond the brief (smells, patterns,
   better ways) already flows to the lead's ledger — growth treats these as CURRICULUM, not just debt:
   a recurring observation class is a def-delta or kit-rule candidate.
4. **Incidents (the hard teacher).** Every incident leaves TWO artifacts by law: the repair + a new
   guard (GUARDRAILS maintenance law) — and now a third: the memory entry that names the lesson.

## GROWTH NOTE — the return-packet section (append when a run taught something craft-level)

```
## Growth note
LESSON: <the craft lesson, one line — technique/failure-mode/benchmark insight>
EVIDENCE: <where it showed (file/run/incident)>
LADDER: <memory-only | DEF-DELTA proposal | KIT-RULE proposal | MACHINE-GATE proposal>
DEF-DELTA (if proposed): <the exact sentence to add/sharpen/remove in my definition — the lead ratifies>
```

## Def evolution — propose, ratify, version (the "better version of myself" mechanism, made safe)

- **An agent NEVER edits its own definition.** Identity is protected the way prod is protected.
- An agent that has outgrown its def **proposes a DEF-DELTA** (sharper reference class, a new duty
  nuance, a named failure mode, a retired stale rule) — in the growth note.
- **The lead ratifies**: checks it against doctrine (no bloat — a def grows only when the lesson has
  RECURRED or is load-bearing; one-run enthusiasm stays in memory), applies the edit, and git carries
  the version history — every def's evolution is reviewable (`git log .claude/agents/<name>.md`).
- The same path DOWNWARD: a def rule proven stale/wrong is removed the same way. Growth includes
  shedding.

## The lead's growth duties (folds into the duty stack — D5)

- **At every stage close:** read the growth notes + Brief-quality self-reports of the stage's runs;
  ratify/reject def-deltas (with the reason on record); promote twice-seen lessons up the ladder;
  prune what aged out. Growth review is event-driven (stage close), never calendar ceremony.
- **Graduation discipline:** memory = personal experience · def = proven identity · kit = team law ·
  hook/fitness = machine truth. Promote at the pace of RECURRENCE, demote at the pace of staleness.
- **Measure sharpness, honestly:** the instruments already exist — the token log (is the same ground
  re-read less?), the ledger (do the same defect classes recur less?), brief-quality reports (is
  judgment exercised growing?). A team that is learning shows it in these numbers without being asked.

## Boundaries (so growth never becomes drift)

- A def-delta must never contradict the hard floor (laws, hooks, work-protection, honesty) — those
  change only by the owner's word.
- Growth notes are DISTILLATE (1–5 lines), not diaries; the memory-home guard's ceilings apply.
- No agent proposes deltas to ANOTHER agent's def directly — cross-agent lessons route through the
  lead (one ratifier, no identity drift).
- The kit stays agnostic: project-flavored lessons go to project memory/context, portable craft goes
  to kit — the same purity law as everywhere.
