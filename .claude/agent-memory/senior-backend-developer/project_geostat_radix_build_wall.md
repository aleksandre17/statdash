Rebuilding the **geostat** dev/prod image at branch tip (fix/portal-review-notes-2026-07-10 `52e5e02`) FAILS: `Rolldown failed to resolve import "radix-ui" from platform/packages/react/src/components/ui/select/Select.tsx`. This is a SECOND, previously-hidden geostat blocker AFTER the CSS `@layer components → sd-components` fix (52e5e02) that the task premise thought was "THE" blocker.

**Root cause.** `packages/react/src/index.ts` barrel hard-exports the radix-based `Select`+`MultiSelect` (`export { Select } from './components/ui/select/Select'`). geostat's `vite.config.ts` aliases `@statdash/react` to SOURCE and, under `.npmrc shamefully-hoist=false`, must alias every peer of the source-bundled packages to the app's own copy — but it auto-derives that list ONLY from each source package's `peerDependencies` (`sourcePackagePeers()` reads `pkg.peerDependencies`). `radix-ui` is declared in `packages/react/package.json` as a regular **`dependencies`** entry (`"radix-ui": "catalog:"`), NOT a `peerDependencies` → it gets NO app-level alias → the production rolldown build can't resolve the source file's bare `import 'radix-ui'`. geostat importing the barrel drags `Select.tsx` into its graph even though geostat's UI doesn't use it.

**Why PANEL is unaffected (proven 2026-07-16).** The panel runs the Dockerfile `development` (vite-dev) target — no rolldown production build. vite-dev resolves `radix-ui` at runtime from `packages/react/node_modules/radix-ui` (radix IS materialized there by the strict-pnpm filtered install — verified in the running panel; the baked barrel already exports Select). Panel rebuild+recreate at 52e5e02 succeeds and serves `Select.tsx` 200. So the wall is geostat-**production-build**-only.

**Why the local proof missed it.** Root `statdash-platform` `build` = `pnpm --filter ./apps/geostat run build` → geostat's own `build` = `tsc -b && vite build`. If the "green local" proof (`pnpm --filter national-accounts build`) didn't exercise the geostat vite build (or ran with a hoisted/full install where radix resolves), the source-alias resolution gap never surfaced; only the Docker per-app filtered-install build exposes it. Same CLASS as [[project_package_build_resolution]] (source-aliased packages hide undeclared/mis-scoped deps that a real bundled build exposes).

**How to apply (FRONTEND/react-specialist owns the fix — do NOT push unilaterally mid-deploy on the shared branch):**
- **Fix options, cleanest first:** (1) Move `radix-ui` (and audit `cmdk`/other barrel-runtime deps geostat's graph reaches) from `dependencies` → `peerDependencies` in `packages/react/package.json` so geostat's auto-alias picks it up — needs a pnpm lockfile regen. (2) ISP fix: split the radix UI components out of the eager barrel into a subpath export (`@statdash/react/ui`) that only panel imports, so geostat's barrel import never drags radix in (aligns with barrel-vs-explicit + strict-SOLID feedback) — geostat doesn't use Select. (3) Add `radix-ui` to geostat's vite.config peer-alias list explicitly (pure build-config, NO lockfile change) — quickest unblock but leaves the peerDeps-derivation gap latent for the next such dep.
- A geostat rebuild at this branch tip WILL keep failing until one lands. The failed build tags nothing → the running geostat stays on its last good image (no regression). Report + route to frontend, don't gamble a working geostat image.

**⚠ RECURRENCE #2 — apexcharts (2026-07-16, blocked the `01d101d` dev redeploy).** The CLASS rule: **any new RUNTIME bare import inside a source-bundled `packages/*` file must be a `peerDependency` of that package** (+ direct dep of geostat & panel). Type-only imports (`import type`) are erased and safe with devDeps — the trap fires the first time a runtime import lands. Instance: `079524b` added `import ApexCharts from 'apexcharts'` to `packages/plugins/panels/chart/default/utils/apex/cartesian/brush.ts` (the rangeSlider brush UMD-global seam) and put `apexcharts: 3.54.1` in plugins' **devDependencies** — one field wrong; local vitest + panel vite-dev pass (devDeps materialize locally), only the Docker filtered-install rolldown build fails (`Rolldown failed to resolve import "apexcharts"`). Fix (frontend owns, mirror c286cef): `apexcharts` → plugins `peerDependencies` (apps already declare it: geostat `^3.45.0`, direct dep) + keep the devDep pin for plugins' own tests + lockfile regen. The geostat Dockerfile comment (apps/geostat/Dockerfile:50-54) already names apexcharts as an expected peer — the manifest just never caught up. **Abort/restore discipline proven:** a partially-successful `docker-compose build api geostat panel` (api built, geostat died) STEALS the `statdash-api:dev` tag from the running container's image → on abort, retag the running image back (`docker tag <running-id> statdash-api:dev`) or a later `up -d` silently ships the mixed-SHA api. Also revert the worktree to the deployed SHA (panel src is bind-mounted from it — verify `git diff <deployed>..<target> -- platform/apps/panel/src` is EMPTY before judging the checkout harmless). Dev line stayed at 41487db, zero downtime.

**RESOLVED + verified — instance 1, radix-ui (2026-07-16, statdash-dev, commit c286cef on `fix/portal-review-notes-2026-07-10`).** Frontend landed option (1): `radix-ui` moved to `peerDependencies` of `packages/react` (+`devDependencies` catalog for its own tests); geostat+panel declare it as a direct dependency; lockfile regen included in the same commit. Rebuilt `statdash-geostat:dev` (image `cea2570f6ec0`) from a CLEAN filtered-install Docker build — `vite build` production/rolldown resolved `radix-ui` with zero errors, new hashed bundle (`assets/index-6_63ZraP.js`, was `index-D7mdtAx8.js`). Also rebuilt `statdash-panel:dev-watch` (`2fe93e4bc397`) to keep the tier on one SHA (panel's Dockerfile target is the vite-dev `development` stage, not the rolldown production path — see [[project_dev_panel_livewatch]] — so it was never blocked by this wall, but a source COPY rebuild is cheap and keeps commit parity). Both containers recreated `--no-deps --no-build --force-recreate`; api/pg containers untouched (uptime unreset). **⚠ Worktree gotcha hit during this deploy:** a linked git worktree (`/tmp/statdash-dev-line`) has its OWN `FETCH_HEAD` pseudoref, separate from the main clone (`/tmp/statdash-build`) even though they share one `.git` — fetching in the main clone does NOT update `FETCH_HEAD` inside the worktree. `git checkout --detach FETCH_HEAD` *inside the worktree* silently lands on the worktree's stale prior fetch, not the sha just fetched elsewhere. Always advance a worktree via the explicit target SHA (`git checkout --detach <full-sha>`), never bare `FETCH_HEAD`, when the fetch happened in a sibling worktree/clone.
