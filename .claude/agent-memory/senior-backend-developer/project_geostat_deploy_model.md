geostat-kit has TWO server-side docker build layouts feeding the SHARED `gen_server_compose.py`, plus a node-vite limitation.

**Why:** the kit builds images server-side from uploaded files; where the upload step lands the context vs Dockerfile differs per stack, so the emitted `build:` must differ.

**How to apply:**
- `gen_server_compose.py` takes `--build-layout {jar|context-dir}` (default `jar`, preserves java-boot). `jar` → `context: .`, `dockerfile: Dockerfile` (java-boot `upload.sh` scp's app.jar+Dockerfile to `$rp`). `context-dir` → `context: ./context`, `dockerfile: ../Dockerfile` (node-api `node-upload.sh` rsyncs the pnpm workspace to `$rp/context/`, Dockerfile to `$rp/Dockerfile`). The layout is declared by the upload step via `export SERVER_BUILD_LAYOUT=...` which `server-compose.sh` reads.
- node-api + node-vite apps build from the pnpm workspace root (`platform/`), NOT the module dir — they need pnpm-lock + `engine/*` siblings + the catalog:/workspace:* protocols (npm cannot resolve these; use pnpm + corepack). Per-module compose build context = `../../` from the app dir.
- RESOLVED 2026-06-28 (was KNOWN GAP): node-vite `deploy.ps1 remote` now archives the WORKSPACE ROOT (`Get-NodeWorkspaceDir` → `Push-Location $WORKSPACE_ROOT; tar … .`) and lands it server-side under `$DEPLOY_PATH/context/<relModule>`, the same `context-dir` layout node-api uses — so the remote container build CAN see the pnpm workspace + engine/ siblings. The earlier limitation (tarred only `$ROOT` module dir) is fixed in the driver. NOTE: this per-module kit driver path is NOT what deploys the live demo — see [[live-deploy-mechanism]] (server git clone + prod-only compose).
