**⚠ CORRECTION 2026-07-01 (proven on real Postgres, SSH staging):** the "fresh V1→V3x deterministic" claim below is WRONG past V32. A truly FRESH DB flyway run FAILS at **V33__demo_classifier_data** with `V33: expected >=11 regions parented to _T, found 0`. V33 is a CORRECTIVE migration that stamps hierarchy/display onto `geo` R2..R12 + `account` + `sector` members that must ALREADY EXIST — but those members are seeded by **canonical ingest** (REGIONAL_GVA/ACCOUNTS workbooks via the api), which runs AFTER flyway in the boot order. At V32 a fresh DB has only geo `GE,GE-KA,GE-TB` + measure(6)+time(6); no account/sector/aggregates/regions. So flyway (migrations-only, all lanes) canNOT reach V33..V38 on a fresh volume. The LIVE prod DB has V33..V37 only because ingest data pre-existed from EARLIER incremental deploys (migrations were added on top of an already-populated cube). This is a real migration-ordering defect: V33 has a runtime data dependency on a step that comes later in the deterministic-bring-up. To provision a fresh DB to V38 you must interleave: migrate → api+ingest (populate classifiers) → migrate the rest — the single-flyway-step prod compose does not do this. To PROVE V38 in isolation, restore a live pg_dump (V37, read-only on live) into an isolated DB, then flyway-advance V37→V38 (V38 only needs concept_scheme/metadataflow/dataset, not the V33 region precondition). Live DB was at **V37** on 2026-07-01 (V38/agency not yet applied to prod). Related: [[live-deploy-mechanism]].

A fresh prod `docker-compose -f ops/compose/docker-compose.prod.yml up` is now deterministic + correct WITHOUT a manual post-step (fixed 2026-06-26).

**Boot order:** postgres healthy → flyway (V1→V34, STRUCTURE only) → api healthy → `ingest` one-shot (canonical workbooks → pipeline) completes → geostat/panel start.

**The 3 load-bearing pieces:**
1. **V34__gdp_dsd_approach_align.sql** — forward migration widening GDP_ANNUAL's DSD from V5/V7's 3-dim `[measure,time,geo]` to the canonical 4-dim `[measure,approach,time,geo]` (series key `{measure,approach,geo}`). The migration form of the live `?datasetVersion=` mint. V5/V7 stay immutable (never edit an applied migration). Guard: refuses to widen if pre-existing 3-dim GDP facts exist (fail-fast vs V22 set-equality trigger); post-condition asserts the final dim order. On a fresh DB GDP has 0 facts at migrate time → guard passes.
2. **R__ neutralized** in BOTH flyway lanes — prod `docker-compose.prod.yml` AND infra `ops/compose/infra/services/flyway.yml` now `-locations=filesystem:/flyway/sql` only (dropped the `/flyway/seed` mount+location). R__ is stale 3-dim and would fail V22 against the 4-dim DSD. R__ file is NOT deleted (kept for CI/restore-net role) — just unmounted.
3. **`ingest` one-shot** (alpine + curl/jq) runs `ops/scripts/ingest-canonical.sh` (promoted from `work/ingest-canonical-e2e.sh`) — POSTs `DATA/canonical/*.xlsx` through `POST /api/ingest/canonical` + publish FSM. Mounts the script + `DATA/canonical` (`../../DATA/canonical` from ops/compose = repo root). Auth from `.env.prod` ADMIN_*. Idempotent (409 ALREADY_PUBLISHED → skip). Bounded retry (deploy.restart_policy max_attempts 3). geostat/panel gate on `service_completed_successfully`.

**Why ingest works on fresh DB (preconditions all hold):** datasets published (V28 backfill), ka+en locales active (V13), GDP DSD 4-dim (V34 → routine compat, no version mint), approach codelist (_Z/PROD/…) publishes before GDP facts (FSM order). Anchor: GDP 2010 total (measure=gross-domestic-product-at-current-prices, approach=_Z, geo=GE) ≈ 22148.65.
