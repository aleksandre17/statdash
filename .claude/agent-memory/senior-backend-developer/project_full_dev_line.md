`ops/compose/docker-compose.dev.yml` (COMMITTED on feature branches) — a THIRD parallel line beside prod ([[live-deploy-mechanism]]) and staging, sharing NOTHING: project `statdash-dev`, net `statdash-dev-net`, volume `statdash-dev-pgdata`, containers `statdash-dev-*`, images `statdash-*:dev`, ports **pg 5458 / api 3011 / geostat 3012 / panel 3013**. Full 5-service stack: postgres + flyway + api + geostat + panel, all BUILT from the branch. Mirrors `docker-compose.staging.yml`'s isolation pattern but is a complete line (own api+db), not the panel-only preview.

**Why:** owner wants a complete dev environment on the DEV server, fully isolated (own infra/db, shares nothing with prod OR staging), built from the current branch — same shape as the prod line. Supersedes and FOLDED IN the panel-only `docker-compose.dev-panel.yml` (3010, staging-api-backed) — see [[isolated-dev-panel]].

**How to apply — key design + deploy facts:**
- **api net-alias `statdash-api`** on statdash-dev-net → the baked geostat/panel nginx `proxy_pass http://statdash-api:3001` resolves to statdash-dev-api (single-origin, `VITE_API_URL`/`VITE_API_STATS_URL` empty = relative `/api`). Alias lives ONLY on dev-net (never prod/stg nets).
- **flyway `-target=32` is LOAD-BEARING on a fresh volume.** Without it flyway runs V1→V38 and DIES at V33 (V33__demo_classifier_data needs region/account/sector members that only exist post canonical-ingest; and V33 < V34-the-GDP-4-dim-widen). Staging's committed file omits -target and only survives because its DB was migrated incrementally, not fresh. A fresh dev/prod volume MUST cap at 32. See [[fresh-provision-canonical]].
- **DATA LOADED 2026-07-11 (was empty); REGIONAL deduped 2026-07-16.** The dev DB is at flyway **V38**; current anchor **REGIONAL_GVA=1650** (post slug-vs-code surgical dedupe, see [[dev-seed-reseed-mechanism]]; was 1665 at load: REGIONAL 1665 + ACCOUNTS 415 + GDP 399 = 2479, 4-dim GDP), loaded via the ADR-035 interleave — but ONLY after hand-applying the `claimed_at` (V37) pre-condition, because `bringup-fresh.sh` is defective as-committed. See [[adr035-interleave-defect]]. The line's containers were NOT recreated (data loaded against the running dev DB/api). The default fresh-boot below still lands empty gold at -target=32.
- **NO ingest one-shot, EMPTY gold on FRESH boot (by default).** A fresh volume cannot deterministically load canonical data (the V33/V34 ordering defect + the claimed_at trap). Gold starts at `stats.observation`=0; the config-tier (bootstrap/site_config/provisioning/auth, all in `config.*` present by V32) is FULLY live — so the AR-49 metric-first panel authoring surface works; geostat map/KPIs render their shell without data. To add real data later: interleave migrate→ingest REGIONAL+ACCOUNTS→migrate V33/V34→ingest GDP→migrate V35-38 (unproven, out of the safe line scope).
- **Throwaway secrets** mirror the staging env contract (env.ts fail-fast: JWT/EMBED ≥32, ADMIN_PASSWORD ≥8): `dev_throwaway_*`, ADMIN admin/dev_admin_pw_123.
- **Deploy recipe (server-native, image-based, no rsync, prod/stg-safe; RE-PROVEN 2026-07-16 @ 41487db):** push feature branch → on server `cd /tmp/statdash-build && git fetch origin <branch>` (fetch does NOT touch the staging clone's 2 uncommitted tracked mods) → worktree ALREADY EXISTS: `git -C /tmp/statdash-dev-line checkout --detach <FULL sha>` (per-worktree FETCH_HEAD → explicit sha; NEVER checkout the branch on /tmp/statdash-build — the dev line builds AND panel-src-mounts from the worktree, and the clone carries staging's 2 uncommitted mods) → `cd /tmp/statdash-dev-line/ops/compose && DOCKER_BUILDKIT=0 docker-compose -f docker-compose.dev.yml -p statdash-dev build api geostat panel` (cache OK — the src COPY invalidation forces a real vite build; confirm new `assets/index-<hash>.js` in the build log) then `up -d --no-deps --no-build --force-recreate api geostat panel`. Panel bind mount = `/tmp/statdash-dev-line/platform/apps/panel/src` (compose default resolved from the worktree) → the checkout itself updates the mounted src; no DEV_PANEL_SRC override needed. Provisioning is idempotent: an api recreate with an UNCHANGED provisioning JSON appends NO page_version rows (bump only on real config delta — attribute deploys by `created_at`). Host uses hyphenated `docker-compose` v5.x. `:dev` tags never collide with prod/stg.
