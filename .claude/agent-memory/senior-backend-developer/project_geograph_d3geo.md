The geograph map is a DECLARATIVE d3-geo SVG choropleth (packages/plugins/nodes/geograph/default/components: GeoMap.tsx + projection.ts + choropleth.ts), which replaced Leaflet to kill the hidden-container blank-map class (geometry from DATA, no DOM measurement). DOM: `svg.geo-map__svg` (data-fitted viewBox) + `path.geo-map__region` with real `d`, `data-selected` (fill `#e8a33d` amber), `data-occupied` (fill `#dc2626` red). Two defects that passed CI + a DOM-metric verify but FAILED on real prod (2026-07-03):

**1. Build-resolution (fixed in 02e3dc3).** The rewrite made `d3-geo` a regular `dependency` of `packages/plugins` — the FIRST external (non-@statdash) regular dep of a source-bundled package. Under `.npmrc shamefully-hoist=false`, apps bundle `@statdash/*` from SOURCE and can ONLY resolve bare specifiers that `apps/*/vite.config.ts` bridges: it reads the UNION of each source package's `peerDependencies` and aliases each to the app's own `req.resolve(id)` copy. A regular dep has NO bridge → clean Docker `vite build` fails `Rolldown failed to resolve import "d3-geo"`. **THE CONTRACT: any external lib imported from packages/{plugins,charts,react,...} source must be a `peerDependency` of that package AND a direct `dependency` of apps/geostat + apps/panel** (leaflet followed this; d3-geo didn't). Fix: d3-geo → optional peerDep of plugins + direct dep of both apps; also dropped the dead leaflet/react-leaflet trio (no source imports them post-rewrite). Local `pnpm --filter ./apps/geostat exec vite build` reproduces the clean-Docker resolution (a dirty root node_modules can MASK it — see [[project_package_build_resolution]]).

**2. Winding-order projection bug (fixed in ce8753d).** The map rendered as ONE SOLID RED SQUARE (every region's path bbox = the full 0..1000 box), NOT distinct polygons. Root cause: d3-geo is SPHERICAL — ring winding decides interior. The live `apps/geostat/public/data/georgia-regions.geojson` ships CLOCKWISE exterior rings, so d3-geo reads each region as "whole sphere MINUS the region": `geoArea(feature) ≈ 4π (12.566)`, and `geoMercator().fitSize` collapses all of Georgia to a ~19px dot while each path fills the frame. Leaflet tolerated it (PLANAR, winding-agnostic); d3-geo does not. Fix in projection.ts: `normalizeWinding()` before fitSize — rewind (reverse every ring) any feature with `geoArea > 2π`. Verified: per-feature geoArea → ~1e-4, regions occupy distinct sub-areas. **The `projection.fitness.test` fixture was correctly wound (CCW) so it never caught this** — the classic fixture-vs-real-data gap; added a winding-agnostic assertion (CW rings must project identically to CCW; no region may fill the whole frame).

**LESSON (both defects): a DOM-metric verify is NOT enough for a map — `13 real paths, non-M0 0, non-zero bbox` was all GREEN while the map was a solid red square. READ THE SCREENSHOT. `getBBox` per path exposed it (every path = full box); `geoArea` confirmed the 4π inversion.**
