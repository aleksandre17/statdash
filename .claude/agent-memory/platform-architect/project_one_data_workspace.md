**DU4a — value-cell `source` variant (DESIGN AUTHORED 2026-07-20, ADR-046 Addendum 4 ACCEPTED, DU4 precondition).** The 4th `SourceStep` union member: the enumerated store-aware VALUE-CELL point read `{op:'source', over, code, coords?, at?, grain?, rollup?, clamp?}` — structurally discriminated by bare `over` (mirrors metrics/query/rows presence-discrimination). **DU4c/d — ADR-046 Addendum 5 (explicit-cells fold, I AUTHORED 2026-07-20).** ratio-list + row-list fold via ONE unified `cells` head — a 5th `SourceStep` member (`{op:'source', cells:ExplicitCell[], enrich?, at?, grain?, rollup?}`), discriminated by `cells` presence, the MEASURE-axis sibling of Add.4's `over`-form (both value-cell family). ExplicitCell = superset `{code, denom?, label?, color?, negate?, isTotal?}`; each kind fills its subset (ratio-list: denom always, no enrich; row-list: denom=pctOf, enrich:true). Decisions on the 3 open Qs: (1) ONE shape not two — kinds diverge only in pure TAIL; (2) enrichment = HEAD-level `enrich?` flag (uniform per spec), NOT per-cell — cell owns label/color VALUES, head owns policy; (3) differing field-sets (measure/pct) = pure tail arithmetic + presence-conditional derive/select. HEAD does store reads ONLY via extracted shared `readExplicitCells` (storeVal + storeVal(denom)→`_denom` + storeObs enrich + tagLocaleString + literal passthrough) — BOTH head arm AND legacy resolvers call it (Strangler seam, byte-identity by construction, FF-EXPLICIT-CELLS-READ-SHARED). Ratio/negate/pct = desugar-emitted tails. Watch: row-list pct is UNGUARDED (0 pctOf→Infinity), condition = "pctOf declared"=`_denom` present; `select` must be PRESENCE-PRESERVING (no null-fill) — flagged to confirm. Multi-code growth (DU4e) does NOT fold here — via `metrics`+calc-metric browse (Add.2), NOT byte-identical by construction (governed vs raw-obs label/color), own proof needed. FF-ALL-KINDS-SHAPED gap after Add.5 = multi-code growth ALONE.
