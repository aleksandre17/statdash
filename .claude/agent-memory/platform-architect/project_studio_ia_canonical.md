`docs/architecture/proposals/SPEC-studio-ia-canonical.md` (I am sole author, 2026-07-12). Resolves **Fable ROOT-2** (the IA) — ROOT-1/3 (object model) are the ADR-041 port phases; this is the authoring-IA PROJECTION of that same port, not a fresh UI.

**Diagnosis (verified in code on feat/ar49-m0-metric-first-authoring):** the Studio has TWO fighting authoring mental models — Model A the contextual right dock (canonical, ~80% built) vs Model B a modal left rail of six peer surfaces (insert·data·layers·pages-site·style·model, `rail.ts`) where "where things live" is a destination you navigate to. Three named coupling smells: (1) `nodeContextEditors['filter-bar']` = a type-keyed dock bridge (the ADR-038 anti-pattern; now redundant — controller already routes parts through `enumerateParts`/`getPartSource`); (2) chrome is a separate selection species (`chromeSel` + `ChromePalette` list in PagesSiteSurface + `ChromeInspectorPanel`) — NOT canvas-selectable (`CanvasOverlay` renders node/drop/part frames, no chrome frames); (3) two authoring homes (Placement Law `relocated-surface` sends site-scope left, element/page right). Plus: RightDock keeps a persistent `Element|Page` Tabs switch (lines 187-197) so page-config bleeds into every element context (owner #3); palette over-shows the whole registry when nothing selected but the page root only accepts `section` via auto-wrap → owner's "blank page only section."

**KEEP (already canonical):** dock section registry (`builtins.tsx`), registry-driven palette (`paletteEntries`), Part-port controller (`useCanvasController` — parts via port already), placement law (scope×weight, derived), URL-as-SSOT routing, focus-view escalation.

**Proposal — canonical set:** Canvas · Left Navigator (Add|Layers, Webflow factoring) · Right contextual Inspector · thin Top bar + TWO summoned project workspaces (Theme, Data-model demoted from rail peers). Right dock = `renderContract(projectContract(selection))` over ONE PartAddress; page-config ONLY when selection empty/page (remove the Page tab); Style/Data/Visibility/Events = declared-facet sections; delete the filter-bar bridge. Palette = `registry ∩ (page-accepts ∪ canonical-wrap-reachable)` — blank page honestly offers every block a section can hold. Chrome → canvas-selectable part of a `site-frame` element (ROM R4 / ADR-041), one selection→one projection.

**Migration (Strangler, felt-impact first):** S1 dock purely contextual (remove Page tab) · S2 palette blank-page fix · S3 delete nodeContextEditors bridge · S4 chrome-on-canvas (interim bridge to existing selectChrome/ChromeInspectorPanel) — all apps-only, reversible, ship now. S5 collapse the rail (Insert+Layers→Navigator; demote Data-model+Style to top-bar) — **owner sign-off (high-visibility)**. S6 selection-triple→one PartAddress + chrome-as-part → delete chromeSel/ChromePalette/element.chrome branch — **rides ADR-041 Ph.3+Ph.6 (Ph.6 one-way, R2-gated), owner sign-off**.

**How to apply:** decision-grade; owner (away) picks direction + routes. Do NOT touch work/ board or ARCHITECTURE-REGISTRY.md (lead registers from my report). Loose-coupling proof: every surface reads port/registry, names NO concrete type; guards FF-NO-EXTERNAL-SPECIAL-CASE→[] · FF-PALETTE-CONTEXTUAL · FF-ONE-PART-GRAMMAR · FF-ONE-SELECTION-ADDRESS.
