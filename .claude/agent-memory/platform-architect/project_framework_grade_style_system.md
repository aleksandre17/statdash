Owner's vision ("elevate style-extension to framework level; bind style to ANY plugin element's inner tags; adopt best concepts from a known framework maybe Tailwind; make it Constructor-authorable") resolved in `docs/architecture/proposals/DESIGN-framework-grade-style-system.md` (design-only, my authorship). **This EXTENDS Z's `DESIGN-responsive-composition.md`, not competes** — Z owns the cascade (`resolveStyle` tiers + `@layer`); this owns the vocabulary+reach flowing through tier ③.

**Core finding:** `NodeStyles` (`styles/src/types.ts:111`, `ResponsiveVal<T>`) is ALREADY the winning primitive — a Chakra/Theme-UI `sx` responsive style-prop object (Builder.io/Plasmic lineage). Missing 3 framework-grade properties: (1) NOT token-constrained at the authoring boundary (`ColorValue=string`, off-scale hex/px legal); (2) can't reach inner PARTS (header `<img>`, chart title — `PageHeaderShell` renders internal React, config has no handle); (3) not Inspector-authorable (style=raw JSON blob only today, the AD-1/AD-3 authoring-lag). Add exactly those 3 → framework-grade with NO new runtime CSS engine.

**Framework-concept adoption (named):** Theme-UI/Chakra/Styled-System = the style-prop object (HAVE IT). W3C-DTCG + Style Dictionary + Tailwind `theme()` = constrained token vocab (HAVE IT: `tokens.css` + `TOKENS_CATALOG` self-describing catalog + `[data-tenant]` Tier-2 rebind spine). **CSS Shadow Parts / Radix `data-part` / Panda-vanilla-extract slot-recipes = named PARTS — THE one new capability**: promote `styleKeys` (currently private BEM SSOT: `HEADER.logo`, `SECTION.body`) to a PUBLIC declared `parts` manifest in `meta`, emit `data-part`, resolve `view.parts.<p>: NodeStyles` per part through the SAME `applyNodeStyles` (each part = a nested styleable leaf of Z's cascade). Tailwind = adopt the CONSTRAINT DISCIPLINE (bounded scale, `[13px]` escape), REJECT class-strings-in-config.

**Constructor answer (owner's open Q):** new `PropFieldType: 'style'` → StyleField control (property-groups × responsive-breakpoint-tabs × token-pickers), driven by the ALREADY-DECLARED-BUT-UNCONSUMED `enum-ref source:'tokens'` (`prop-schema.ts:66`) resolved against `TOKENS_CATALOG`, filtered by `TokenGroup`; part-selector dropdown from the declared `parts` manifest. Flows through `describeApp()` (Cluster ① authoring SSOT). Token serialized form STAYS `var(--*)` (zero migration, already themeable) — the constraint lives in the picker, NOT a new `{token}` wrapper (deferred D-TOKEN-WRAPPER); round-trip via `cssVar → token key` reverse-lookup on the catalog.

**Rejected:** raw-Tailwind-classes-in-config (breaks no-code + agnosticism + lossless round-trip + `[data-tenant]` theming + Law 2 — utility classes solve hand-authoring, we build visual-Constructor-over-data, opposite requirements); unbounded-arbitrary-CSS-as-default (the constraint IS the value — finite pickers, total theming, uniform-by-construction sections); build-time-atomic-engine Panda/vanilla-extract (Golden Hammer, fights runtime per-tenant theming — the runtime var+flag `applyNodeStyles` is canonical); per-node bespoke forms (shotgun surgery vs generic StyleField).

**Section-uniformity = constraint-by-construction** (can't drift off a scale you can't type — the Tailwind insight applied to no-code), enforced by fitness functions not a style guide.

**Strangler:** S0 (token-picker StyleField, activate `source:'tokens'` + reverse-map) → S1 (surface `view.styles` on section/panel/chart, rides Z tier ③) → S2 (parts mechanism: declare/emit/resolve, populate 1st REAL consumer only — no empty cathedrals) → S3 (constraint FF warn). **MVP = S0+S1 (no parts yet)** — closes the biggest real gap. 6 FFs: NO-RAW-CLASS-IN-CONFIG, PARTS-DECLARED (no-dead-part, mirrors Z's NO-DEAD-CAPABILITY), STYLE-ROUNDTRIP, TOKEN-CONSTRAINED (warn), THEME-REBINDABLE, TOKEN-PICKER-COVERS-CATALOG.

**How to apply:** treat as the binding target for style-authoring + inner-element-styling work. Style is DATA (token-bound NodeStyles object), never a class string. New stylable inner element = one `parts` manifest entry + `data-part` emit + per-part resolve, never plugin-internal-only. Ties to [[project-responsive-composition-model]] (Z's cascade) and [[feedback-maximal-orthogonality]] (parts extend reach along the style axis, not a new duplication axis).
