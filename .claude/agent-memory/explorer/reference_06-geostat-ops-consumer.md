## What Kit Reads From ops/

**`compose/catalog.json`**
- Templates (docker-compose service blocks with {var} placeholders)
- Targets (which templates в†’ which docker-compose files)
- Features (worker enabled? custom flags?)
- Kit calls: `compose/build.py` в†’ load_catalog() в†’ render each target

**`config/*/.env.example`**
- Kit references paths (e.g., `ops/config/backend/.env.dev`)
- CI: Kit helps scaffold real `.env.dev` from `.env.example`
- Not generated by kit; consumer provides

**`config/deploy.env`**
- Kit reads: `DEPLOY_SERVER`, `DEPLOY_PROJECT`, `COMPOSE_*` overrides
- If missing, kit uses defaults (DEPLOY_PROJECT = repo folder name)

**`config/ssh/`**
- Kit references in deploy scripts
- Consumer provides actual keys

**`ci/integration-stack.sh`**
- Manifest points to it: `ci.integration`
- Kit calls it from CI pipeline
- Consumer owns full orchestration

**`ci/wait-stack-health.sh`**
- Manifest points to it: `ci.waitStackHealth`
- Kit provides default stub in `kits/geostat-kit/ci/wait-stack-health.sh`
- Consumer extends with actual endpoints

## What Kit Does NOT Read From ops/

- Individual app `docker-compose.yml` files (kit GENERATES these)
- `package.json`, `tsconfig.json`, `build.gradle` (in `apps/`, not `ops/`)
- `.env.dev` real values (kit only checks if file exists; uses it during compose/deploy)

## Consumer Responsibility

1. **Create `ops/config/`** вЂ” all real env values
2. **Create `ops/compose/catalog.json`** вЂ” templates for compose generation
3. **Create `ops/compose/infra/`** вЂ” infra service definitions
4. **Create `ops/ci/`** вЂ” orchestration + smoke tests
5. **Update `geostat.ops.json`** вЂ” manifest that drives everything
