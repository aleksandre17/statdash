As of 2026-06-23 the DB layer is well established (the 2026-06-13 "greenfield, no SQL" note is OBSOLETE). Flyway migrations live in `ops/postgres/migrations/` (V1 extensions → V26 content_constraint; V13 is the latest-numbered i18n pair, note V13/V14 and V19/V20 are out of filename order but Flyway orders by version number). Engine = **timescale/timescaledb-ha:pg16** (CI + all compose files) — so PG16 features (UNIQUE NULLS NOT DISTINCT, etc.) are available; TimescaleDB hypertable on `stats.observation`, LTREE on `stats.classifier`.

**Three schemas:** `config` (pages/nav/data_source/site_config + set_updated_at()), `stats` (gold cube: dimension, classifier SCD-2, dataset, dataset_dimension DSD, observation hypertable, release/vintage V25, content_constraint V26), `stats_stage` (Medallion bronze/silver: submission FSM, obs/classifier/display staging, validation_issue).

**Key invariants to honor:** Law 1 (dimensions are DATA — `stats.dimension` rows, never columns; `dim_key` is generic JSONB). dim_key validated in isolation by the V4 BEFORE trigger (key set == DSD, each value ∈ classifier) and in COMBINATION by the V26 silver gate (not a trigger). No FK targets the observation hypertable (V8). Migration headers carry a full 09 §B risk gate + COMMENT ON everything; additive/idempotent; never edit an applied migration.

**TimescaleDB partition-column rule (V4, final 2026-06-23):** `stats.observation.time_period_date` (the hypertable partition dimension) is a PLAIN DATE that is **WRITER-PROVIDED** — every observation writer supplies it inline as `stats.parse_time_period(time_period)`. TWO mechanisms were tried and both PROVEN to fail on live TimescaleDB: (a) GENERATED STORED — forbidden as a partition dimension (SQLSTATE 0A000); (b) a BEFORE INSERT trigger (`derive_observation_time_period_date`) — TimescaleDB enforces the partition-column NOT-NULL check BEFORE user BEFORE-row triggers run, so the trigger never populates it on INSERT and every insert failed with `NULL value in column "time_period_date" violates not-null constraint`. The trigger + its function were DROPPED from V4 (still never-applied → edited in place). `dim_key_hash` STAYS GENERATED STORED (legal: in the unique index + compress_segmentby, NOT the partition dim). The date rule SSOT is `stats.parse_time_period` — writers call it BY NAME so V9 frequency widening flows in automatically; no writer reimplements it. **All four obs writers fixed:** `ingest/publish.ts` (set-based INSERT…SELECT), `ingest/upsert.ts` (`upsertObservation`), `scripts/seed-helpers.ts` (`upsertObservation`), and the R__ stage-1 gold seed SQL (~2131 rows). **Fitness guard:** `platform/apps/api/src/ingest/observation-partition.fitness.test.ts` (no-DB, source scan) asserts no `INSERT INTO stats.observation` omits `time_period_date` and that each writer uses `stats.parse_time_period` — prevents the NULL-partition bug from returning. Remaining V4 BEFORE-row trigger firing order (alphabetical): `_aa_release_stamp` (V25) → `capture_revision` (V8) → `updated_at` (V4) → `validate_dim_key` (V4). The two live DB-gated tests that insert observations (`routes/cube/cube-profile.test.ts`, `routes/stats/dataset-lifecycle.fitness.test.ts`) were updated to provide the column too.
