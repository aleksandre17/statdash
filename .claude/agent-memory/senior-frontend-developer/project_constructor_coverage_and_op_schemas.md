Constructor V1 slice (schema-driven transform StepForms + the north-star coverage gate). Builds on [[constructor-inspector]].

**PropSchema vocabulary now lives in `packages/core`, not react.** `PropField`/`PropSchema`/`PropFieldType`/`PropFieldSource`/`PropFieldOption`/`PropFieldValidation`/`PropertyGroup` moved to `packages/core/src/config/prop-schema.ts` and are exported from `@statdash/engine`. `packages/react/src/engine/slice-meta.ts` RE-EXPORTS them (`export type { … } from '@statdash/engine'` + a local `import type` for in-file annotations like `PropertyGroup`/`PropSchema`) — so all ~73 `@statdash/react/engine` import sites are byte-identical.
**Why:** a TransformStep op must CARRY its own authoring PropSchema (OCP), the op lives in core, and the arrow forbids core→react. PropField's only external dep is `LocaleString` (core-origin), so core is its correct home. It CANNOT go to contracts (contracts is zero-@statdash-dep; PropField.label is a core LocaleString).
**Gotcha:** when re-exporting a type that's also used in value/annotation position in the same file, you need BOTH `export type { X } from …` AND `import type { X } from …` or tsup DTS build fails with "Cannot find name X".

**Transform-op authoring registry (the schema-driven StepForm seam).** `registerTransformStep(op, fn, schema?)` now takes an optional PropSchema; `getTransformStepSchema(op)` / `listTransformOpSchemas()` added (`step-registry.ts`, exported from engine barrel). Built-in op schemas live in `packages/core/src/data/transform/op-schemas.ts` (catalog-class; bilingual {ka,en} labels) and register WITH each op in `transform/index.ts`. 17 of 18 ops carry a schema; `joinByField` is intentionally schema-less (carries resolved EngineRow[], not declaratively authorable — permanent COVERAGE_TODO).
**Panel render path:** `TransformStepEditor` (`features/data-layer/editors/query/steps/`) models a step as a CanvasNode `{type:step.op, props:step}` and renders it through the EXISTING `<Inspector>` via `transformStepSchemaSource` (a SchemaSource whose getSchema = getTransformStepSchema). Writes back with `setAtPath(step, field, next)` (from `inspector/showWhen`). `op` is never in any schema → discriminant immutable. `StepForm.tsx` dispatch: derive/lookup/sort/filter keep bespoke forms (byte-identical), every other schema-carrying op → TransformStepEditor, schema-less → raw-JSON. `PipelineBuilder` OP_OPTIONS is derived from `listTransformOps()` minus joinByField (was a hardcoded 4); `defaultStep` extracted to `steps/defaultStep.ts` with a factory per op.

**Coverage Fitness #1 gate** (`features/data-layer/coverage.fitness.test.ts`): enumerates from engine SSOT — `listTransformOps()` (true runtime registry) for ops; `DATASPEC_DISCRIMINANTS`/`PARAMDEF_TYPES`/`VISIBILITY_OPS` (new `packages/core/src/config/discriminant-manifest.ts`, compile-time-EXHAUSTIVE tuples via `Exact<tuple,union>` assertion — adding a union member without the tuple is a TYPE ERROR) for the type-only unions. Asserts each member has an authoring surface OR is in an explicit roadmap-keyed `COVERAGE_TODO` allowlist (a surfaced-AND-allowlisted member also fails — keeps the gap list honest; a stale allowlist key also fails). Current TODO: dataSpecs row-list/by-mode/pivot/transform (V2) + custom (PERMANENT); ALL paramDefs (V0); ALL visibilityOps (V4); transformOps joinByField (PERMANENT).

**no-tenant-content fitness gotcha:** `platform/tests/no-tenant-content.fitness.test.ts` bans Georgian script in `packages/*` library code EXCEPT an `ALLOW` catalog-class list. Any new bilingual authoring CATALOG in core (like op-schemas.ts, sibling to spec-catalog.ts) must be added to that ALLOW set. (The react-engine copy of this test only scans packages/react, ALLOWED empty by design.)
