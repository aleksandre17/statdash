apps/{geostat,panel}/vite.config.ts pin EVERY peerDependency of the @statdash/* packages they bundle from SOURCE, so a CLEAN (Docker) build resolves them in-graph. This is DATA-DRIVEN, not a hardcoded peer list.

**Why:** apps build @statdash/* SOURCE via the resolve.alias source entries, so Vite/Rolldown walks packages/* and hits THEIR own bare peer imports from WITHIN packages/* — `react-router-dom` & `i18next` (from packages/react), `react-leaflet`/`leaflet`/`react-apexcharts` (optional, from packages/plugins), plus react/react-dom. With .npmrc shamefully-hoist=false (strict isolation) those peers are NOT linked into packages/<pkg>/node_modules. On a clean install: REQUIRED peers fail with `failed to resolve import "react-router-dom" from .../FilterContext.tsx`; OPTIONAL peers get an empty `__vite-optional-peer-dep:...` stub → MISSING_EXPORT. A dirty local node_modules MASKS all of this; only the clean Docker build shows it. The optional-4 fix was incomplete — the error just moved to the next unresolved required peer. Do NOT convert any peer to a hard dep; keep shamefully-hoist=false.

**The mechanism (each config, local helper — no shared util exists, established pattern is per-app duplication):**
- `SOURCE_PACKAGE_DIRS` = the 7 packages aliased to source (contracts, expr, core, charts, react, plugins, styles). `sourcePackagePeers()` reads each package.json's `peerDependencies` and UNIONs the names — the package.json IS the SSOT. Current union (both apps, identical): react, react-dom, react-router-dom, i18next, react-apexcharts, leaflet, react-leaflet (7).
- `peerEntry(id)` (createRequire(import.meta.url)) resolves each to its absolute ESM ENTRY FILE; alias `{ find: /^id$/, replacement: <abs entry> }` (regex so subpath imports like `leaflet/dist/leaflet.css` are untouched).
- `resolve.dedupe = PEER_SINGLETONS = ['react','react-dom','react-router-dom']` (always-required singletons; optional/feature peers don't need single-instance pinning).
- GUARD: peerEntry try/catch on req.resolve → throws a clear `peer "X" is not resolvable... declare it as a direct dependency` instead of emitting `undefined` into resolve.alias (a prior run printed a bare `undefined`). Apps DO declare all 7 as direct deps, so it never fires.
- react-i18next is NOT in the union (not declared as a peer, not imported from source packages) — correctly excluded.

**Gotcha 1 (entry-file, not directory):** aliasing a peer to its package DIRECTORY re-runs Vite legacy main-field resolution → prefers top-level `browser` field. react-apexcharts' `browser` = IIFE bundle with NO `default` export → MISSING_EXPORT. Fix: resolve the explicit ESM entry from exports['.']. react-leaflet does NOT expose `./package.json`, so walk up from require.resolve(id) to the package.json whose name===id.

**Gotcha 3 (subpath imports → RUNTIME "Could not resolve", not a build failure):** the exact `/^id$/` entry alias resolves the BARE specifier but deliberately leaves SUBPATHS (`leaflet/dist/leaflet.css`, `apexcharts/...`) to normal resolution — which, for a peer of a source-bundled @statdash/* package under filtered install + shamefully-hoist=false, has NO resolution. Vite then emits a RUNTIME `Uncaught Error: Could not resolve "leaflet/dist/leaflet.css" imported by "@statdash/plugins"` stub — the build PASSES, the deployed app throws. FIX: a SECOND alias per peer `{ find: /^id\//, replacement: peerDir(id) }` where peerDir = package ROOT dir (the dir containing package.json whose name===id) + trailing `/`. peerAliases is now `flatMap` emitting BOTH aliases, exact `/^id$/` BEFORE prefix `/^id\//` (vite alias = first-match: bare still hits the ESM entry, avoiding the Gotcha-1 browser-IIFE trap; only subpaths use the directory). Extracted `peerPackageDir(id)` shared by peerEntry (ESM entry) + peerDir (subpath base). VERIFY is runtime, not build: `grep -rn "Could not resolve" apps/{geostat,panel}/dist` must be EMPTY + confirm leaflet.css asset with `.leaflet-` rules actually landed in dist — a passing build alone is NOT sufficient evidence. Lint note: typed peerPackageDir's pkg as `Record<string,unknown>` (NOT `any` — no-explicit-any is an ERROR here) and narrowed `pkg.exports` before indexing `['.']`.

**Gotcha 2 (nested export conditions — cost a build when i18next entered the set):** the old entry picker `dot?.import ?? dot?.default` assumed flat string targets. i18next's `exports['.'].import` is itself `{ types, default }` (an OBJECT) → `resolve(dir, <object>)` threw `ERR_INVALID_ARG_TYPE: paths[1] must be string`. Fix: a RECURSIVE `pickExportTarget(node)` that walks condition objects in order ['source','import','browser','module','default'] until it hits a string, then falls back to pkg.module → pkg.main; throws if no string found. Only looks INSIDE exports['.'] (+ module/main) — never the top-level `browser` field, so Gotcha-1 trap stays avoided.

Both builds verified green (geostat entry 68.93 kB; panel built; no failed-to-resolve / MISSING_EXPORT / __vite-optional-peer-dep). typecheck clean, lint 0 errors. Chunk groups (react-vendor/apexcharts/leaflet/i18n/mui/dnd-kit/cmdk) + @statdash/* source aliases + rolldownOptions all intact (ADDED to resolve, not rewritten).
