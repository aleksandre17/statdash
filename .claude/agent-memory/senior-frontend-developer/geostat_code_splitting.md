apps/geostat (the public SDUI renderer site, de-tenanted runner ADR-0028) shipped a single eager Vite/Rolldown entry of 1,254.63 kB (gzip 356.50). It is now route/feature code-split so the boot shell stays small and the heavy renderer graph loads on demand. Mirrors the apps/panel pattern ([[panel-code-splitting]]).

**The single lazy boundary — `src/app/RendererSurface.tsx`:**
- Geostat's whole renderer weight roots in TWO eager static chains from main.tsx: (1) `setupRegistrations()` → `@plugins/panels` (ApexCharts ~540 kB) + `@plugins/nodes` (Leaflet) and (2) `App → LocaleGuard → NodePageRenderer + AppChrome` (@statdash/react engine). BOTH must move behind one dynamic import to clear the entry.
- `RendererSurface.tsx` owns BOTH: it runs `setupRegistrations()` at module top-level AND renders the `SiteProvider` + `Routes` (→ LocaleGuard) tree. App.tsx lazy-loads it via `React.lazy(() => import('./RendererSurface'))` behind `<Suspense fallback={<SuspenseFallback label="Loading…"/>}>`.
- main.tsx keeps `i18next.init()` (synchronous, before createRoot) + theme + createRoot eager; its `setupRegistrations()` CALL was REMOVED (the heavy SLICE/projector wiring now runs at RendererSurface module load). init → register-slices → render order is preserved across the split for the slices.
- App.tsx keeps bootstrap (bootstrapSite), AppSkeleton, registerFormatters + modeRegistry.register (post-boot) eager.

**ORDERING REGRESSION (fixed 2026-06) — store-builders must register EAGERLY, before bootstrap.** The original split moved `registerStoreBuilders()` into the lazy `setupRegistrations` too. But App's `bootstrapSite()` (useEffect) CONSUMES the store-builder registry: `fetchStoreManifest → buildStoreManifest` dispatches each datasource to its registered builder and THROWS on an unregistered kind. The lazy RendererSurface only loads AFTER bootstrap resolves → every store failed to build (`No StoreBuilder registered for kind 'stats'`) → all charts/tables/maps rendered empty on the live deploy. FIX: `registerStoreBuilders()` was split OUT of `setupRegistrations` into `src/bootRegistrations.ts` (`bootRegistrations()`), called EAGERLY in main.tsx BEFORE createRoot().render(). It is LIGHT — `@statdash/plugins/datasources` pulls NO ApexCharts/Leaflet, so the entry chunk is unchanged size-wise. `registerStoreBuilder` is `Map.set` (idempotent, no throw on dup) but the call was REMOVED from setupRegistrations (not double-registered) so the eager-vs-lazy split is explicit, not reliant on overwrite. The lesson: "init → register → render" overlooked that BOOTSTRAP (a third actor) consumes a registry before the lazy register — any registry consumed at bootstrap must be primed eagerly.
**Regression guard:** `src/data/boot-store-registration.test.ts` asserts `buildStoreManifest([{kind:'stats'}])` SUCCEEDS after `bootRegistrations()` WITHOUT importing the lazy setupRegistrations/RendererSurface. Vitest `isolate:true` gives each file a fresh `_registry` singleton, so the pre-fix failure is deterministic (proven: stubbing out the eager call reproduces the exact prod error).

**Test invariant (load-bearing):** the fitness tests (`second-tenant.fitness.test.tsx`, `inner-sidebar-slot-config.test.tsx`) import `setupRegistrations` AND `LocaleGuard` DIRECTLY and call them synchronously — they never load through the lazy RendererSurface module. So `setupRegistrations` MUST stay a plain synchronously-importable function and LocaleGuard a normal export. Do not move registration logic somewhere only reachable async. 12/12 geostat tests pass unchanged.

**Accessible fallback — `src/app/SuspenseFallback.tsx`:** geostat-local (NO MUI — geostat has no MUI dep, unlike panel). role=status + aria-live=polite + aria-busy + aria-label; reuses the existing `app-skeleton` DOM structure (layout-stable, no CLS, brand-neutral). Matches the eager AppSkeleton's aria-label approach (the `.app-skeleton`/`sr-only` CSS classes are referenced but unstyled across the app — pre-existing, a11y carried by ARIA not visuals).

**Vendor chunking (vite.config.ts → build.rolldownOptions.output.codeSplitting.groups):** react-vendor (priority 40), apexcharts (30), leaflet (30), i18n (25). Same Rolldown gotcha as panel: react-vendor MUST be highest priority so react/jsx-runtime lands in the small eager vendor chunk, NOT pulled into apexcharts by includeDependenciesRecursively. VERIFIED via dist/index.html modulepreload list = only rolldown-runtime + react-vendor + i18n (apexcharts/leaflet/datasources/RendererSurface NOT preloaded).

**Result:** eager = index 67.79 + react-vendor 246.73 + i18n 47.28 + runtime 0.69 ≈ 362 kB raw / ~113 kB gzip (all under the 500 kB warning). On-demand = apexcharts 537, leaflet 151, RendererSurface 112, datasources 97. The lone remaining 500 kB warning is the lazy apexcharts chunk (the charting lib itself, behind the boundary) — same accepted residual as panel. i18n is eager because main.tsx statically needs i18next.init() (correct).

**Pre-existing blocker (NOT mine, out of scope):** the shared `tsc -b` (pnpm typecheck / build:geostat) fails on foreign in-flight errors in `packages/plugins/datasources/href-registrations.{ts,test.ts}` (untracked, another dev's mid-edit) — TS2339/TS2322. Whole-program typecheck surfaces them via project refs. Geostat sources are 100% type-clean; verified the split via direct `vite build` (bypasses the cross-package tsc). Do NOT fix href-registrations — it's someone else's uncommitted WIP (scope = apps/geostat only).
