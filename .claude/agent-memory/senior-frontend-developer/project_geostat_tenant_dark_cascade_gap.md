AR (theme-contrast sweep, 2026-07): reported bug was "white hero text on a light card" in dark mode. Two DISTINCT root causes, both in the theming spine — worth knowing apart before touching either again.

**Root cause A — a role that should NEVER flip:** `--color-heading-display` (packages/styles/src/css/tokens.css) had a dark-mode override (`#f0f3fa`, near-white) like every other Tier-2 role. But its ONLY consumer, `packages/plugins/nodes/hero/default/hero.css` `.hero__title`, sits on `HeroCardDef.pageBg` — a per-card, config-authored gradient (`apps/api/provisioning/geostat.provisioning.json`), never on `--color-surface`. That backdrop doesn't participate in the app theme, so the text color must not either. Fix: REMOVE the dark override (both `@media` and `[data-theme=dark]` blocks) and document it as `PINNED_NO_FLIP` — a small, explicit allowlist added to `tokens.parity.test.ts`'s FF-DARK-COMPLETE check (a role legitimately absent from dark, not forgotten). Added a sibling `--color-heading-display-muted` (`var(--blue-700)`) for `.hero__subtitle`, which had been wrongly using the ADAPTIVE `--color-text-muted` on the same non-adaptive backdrop.

**Root cause B — a role that SHOULD flip but the tenant broke the mechanism:** `apps/geostat/src/shared/styles/index.css`'s `[data-tenant="geostat"]` block pins `--color-accent/-hover/-muted/-bg/-chip-border/-secondary` and the trend pair UNCONDITIONALLY — no dark selector at all. `[data-tenant="geostat"]` and `[data-theme="dark"]` are both single-attribute selectors (equal CSS specificity); because `packages/styles`' tokens.css loads BEFORE the app's own tenant stylesheet in the Vite bundle, the tenant rule wins the cascade by SOURCE ORDER regardless of theme — so `--color-accent` stayed pinned at `#0080BE` (tuned for a white surface) even in dark mode. Computed contrast: every accent-text consumer (header/footer links, filter-bar chips, breadcrumb hover, kpi-info-link hover) rendered at only 4.17:1 against the dark surface (`#15151f`) — an AA failure invisible unless you diff both themes side by side. The trend pair (`#2A9D8F`/`#E76F51`) ALSO failed AA in LIGHT mode (3.32:1/3.09:1 on white) — a pre-existing bug unrelated to dark mode, caught by the same sweep.

**Fix pattern for B (reusable):** add `@media (prefers-color-scheme: dark) { [data-tenant="geostat"]:not([data-theme="light"]) {...} }` PLUS `[data-tenant="geostat"][data-theme="dark"] {...}` (both required, kept byte-identical — mirrors tokens.css's own dual-block idiom), with brightened/re-hued values mirroring the SAME light→dark relationship `packages/styles`' default theme already establishes for `--color-accent` (darker→brighter+saturated) and `-muted/-bg/-chip-border` (light pastel tint→dark desaturated tint). Compute via HSL-transform-copy: take the DEFAULT theme's own light/dark hex pair for a role, derive its HSL delta, and apply that SAME delta to the tenant's own hue. Verified numerically (Node `contrast()` scripts), not eyeballed — see the WCAG assertions in `apps/geostat/src/shared/styles/tenant-theme.fitness.test.ts` (new FF-TENANT-DARK-COMPLETE fitness, mirrors tokens.css's FF-DARK-COMPLETE for tenant stylesheets specifically).

**Gotcha worth repeating:** don't assume a tenant stylesheet's dark-mode behavior from reading `tokens.css` alone — a tenant CAN silently override the whole mechanism by pinning a Tier-2 role unconditionally with no dark selector; the bug is invisible until you actually compute contrast against the DARK surface for whatever the tenant pinned. See also [[css_fitness_comment_stripping_gotcha]] for a parser bug found while building this fitness function.
