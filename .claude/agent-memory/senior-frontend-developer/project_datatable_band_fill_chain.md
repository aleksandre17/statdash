A banded table view (AR-8 height band around a chart↔table toggle) can fail to scroll: the table renders at full content height, overflows the band, and the band's `overflow:hidden` clips the bottom rows (no scrollbar). Root-caused live on `/ka/regional` ("GDP — by region" composition table, a 603px table inside a 362px band → not scrollable).

**Root cause (the asymmetry to remember):** node-styles.css hands the band box a definite height and flags `.data-table__wrap { flex:1 }`, BUT `TableShell.tsx` renders an intermediate `<div {...bodyAttrs}>` (title-badges + `<DataTable>` + `<PanelExportBar>`) between the band box and `.data-table__wrap`. That div is `display:block` → (a) the wrap's `flex:1` is INERT against a non-flex parent so the band height never reaches it, and (b) it grows to full table height and overflows the band. The **chart** shell does NOT have this problem because `Chart.tsx` spreads `{...bodyAttrs}` DIRECTLY onto `.chart-wrap` (`<div {...bodyAttrs} className="chart-wrap">`), so the chart's content box IS the band box's direct child and node-styles' inverted-band-leaf rule (`[data-view=visible] > .chart-wrap[data-aspect]`) catches it. The table needs a wrapper (badges/export bar are siblings of the wrap), so its content box sits one level too deep. → **This is table-specific; it does NOT share the ApexCharts NaN-dimension root** (that's a hidden-view re-measure path, different mechanism).

**Two band-delivery shapes both hit this** (see [project_sna_table_is_pivottable] for the SNA/PivotTable dispatch): (1) `view.styles.height` on the **section** → `data-height` on `.section__body`/`.panel__body`, band on an ANCESTOR `[data-view]` slot; (2) `view.styles.aspectRatio` on the **child view** → `data-aspect` on the shell body wrapper itself. Composition `sectors` uses shape 1; comparison `regions-bar` uses shape 2 (latent at 1440px — its ~646px band fits 11 rows, but structurally identical).

**Fix (in `data-table.css`, table-owned, agnostic):** make the element that directly parents the wrap a fill-flex link, selected via `:has(> .data-table__wrap)` (SSOT) under the band's own `[data-view]/[data-height]/[data-aspect]` attributes — reading, never redefining the band (same `:has()` idiom node-styles uses for the chart leaf). Declaration: `display:flex; flex-direction:column; flex:1; min-height:0`. Unbanded tables (`/ka/accounts`) match none of the selectors and keep the `max-height:var(--data-table-max-h,70vh)` viewport from [project_scrollbar_utility_and_table_overflow] — no regression. Bonus: un-clips the per-section export bar (Law 9), previously lost off the band bottom.
