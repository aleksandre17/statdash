ADR-051 DU4 folds `timeseries` + single-code `growth` onto a value-cell `source` head `{op:'source', over, code, coords}` (no `dataSource`). In the workbench (`usePipelineSourceRows`), store resolution is `resolveStore({stores, pageStoreKey: specDataSource(sourceSpec)})`. For a bare measure `code` that is NOT a governed metric, `specDataSource` returns undefined → `resolveStore` falls back to the FIRST store in the live map.

**Verified defect (2026-07-20):** a `timeseries {code:'GDP'}` (GDP lives in the `gdp` cube) opened on the regional-first Model floor rendered `2015…2022 | GDP | 0 | 0` — PointSeriesResolver's empty-sum returns a real **0** (not `MISSING_GLYPH '—'`) because the resolved (regional) store has no GDP cell. Same spec with `code:'GVA'` (in the default store) rendered real values (61291.16…126793.1). So the 0 is a **fabricated value**, exactly what Law 11 ("the canvas never lies … never a fake 0") forbids. Step A newly EXPOSES this latent lie by opening these specs in the grid for the first time.

**How to apply:** When reviewing the value-cell fold / head store-pinning follow-up, require the honest state: a value-cell head whose store can't resolve the measure must render unbound/no-data, never 0. Candidate fitness: assert PointSeriesResolver (or the grid) distinguishes "cell absent" from "cell == 0". Not caused by Step A's panel code — root is the value-cell head's missing store-home + PointSeriesResolver 0-for-missing. Related: [[systemic-findings]].

**F2 FIX — VERIFIED LIVE (2026-07-20, dev :3013 engine at HEAD via docker-cp).** Fix = two arms in `packages/core`: (1) `measureRefs` value-cell arm `if ('over' in head) return [head.code]` so `specDataSource` routes the folded head; (2) honest-missing backstop `noData:'null'` on the PointSeries variant → `desugarTimeseries` sets it, `PointSeriesResolver` reads via new `storeCellAt` (obs-scan null-for-absent), growth omits it (GrowthResolver `?? 0` byte-identity). Verdict PASS: fabricated 0 GONE, cross-store governed metrics route to REAL values, absent coords read honest '—', no regression.

**CRUCIAL nuance the original bug-note (and the F2 task brief) got WRONG:** cross-store routing works ONLY for a GOVERNED METRIC-ID, never a raw cube code. Live proof: `timeseries {code:'gdp.current'}` (the metric-id, MetricDef.dataSource='gdp') → routes to gdp store → REAL 2018=136123.08 … 2025=313794.42. But `{code:'GDP'}` OR `{code:'gross-domestic-product-at-current-prices'}` (raw codes) → ALL years '—', INCLUDING 2025 which provably has data (raw browse showed 104598.14×3 approaches). Why: `resolveMeasureRef(rawCode).dataSource` is undefined (only the metric-id carries the store-home per Law-1/M1 "a metric NAMES its store") → `specDataSource` returns undefined → `resolveStore` falls to the FIRST/regional store → regional lacks it → honest '—'. So raw-code '—' is CORRECT (Law-11 honest, not a lie) — but a re-tester using literal raw `GDP` will see '—' and may misread it as a FAILURE. Always cross-store-test with the governed metric-id (`gdp.current`), not the raw code. Test specs left on dev named `F2-VERIFY *`.
