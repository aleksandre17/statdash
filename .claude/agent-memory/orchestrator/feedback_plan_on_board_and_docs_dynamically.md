**Rule:** a senior lead does not keep the plan in his head or scatter it across chats. He **routes each kind of thinking to its right home**, and uses ALL the instruments the project gives him — DYNAMICALLY, always as the situation is best, never by rigid rule and never as empty ceremony. Plan first, then execute faithfully so there are **no deviations and no misses**; and keep the instruments CURRENT as work lands (a decision made but unregistered, a vision unfiled, a finding un-carded = lost knowledge).

**The instrument set (verified live, 2026-07-11) → what each is for:**
- **`work/` board** (`items/` cards, `BOARD.md` view, `PROCESS.md`) — the OPERATIONAL plan: the ordered pipeline of what-after-what, deps, status. Nothing starts off-board (`11` rule 1); `backlog→ready` is the owner's call.
- **`docs/architecture/decisions/` (ADRs)** — durable DECISIONS with ≥2 rejected alternatives + a fitness function. Any new pattern/structural call → an ADR.
- **`docs/architecture/ARCHITECTURE-REGISTRY.md`** — the CATALOG of architectural visions/initiatives (register a vision here before designing it; the charter's proactive-innovation home).
- **`docs/architecture/BENCHMARK-REFERENCE-PLATFORMS.md`** — the reference-class to benchmark against ("how would Builder.io / Cube / Malloy / Framer solve this?").
- **`docs/plan/`** — roadmaps, phase plans (the strategic arc above the board's tactical cards).
- **`docs/audit/`** — audit boards + findings (MASTER-BOARD, domain-boards, review boards).
- **`docs/knowledge/`** — durable knowledge base; **`docs/patterns/`** — reusable patterns; **`docs/layers/`** — layer specs; **`docs/rules/`** — rules.
- **agent-memory** — cross-SESSION truth (what a doc/board can't carry between sessions).

**Why (2026-07-11):** the owner: *"a big update is coming — we have a work env AND a docs env; we also have registries, and architecture/knowledge/plan/audit and other folders. Approach it dynamically — a professional team writes a plan and executes it so there are no misses. You must KNOW when to use what, like a senior leader, always as the situation is best — not from a rigid rule, not from unnecessary ceremony. Keep this session to session."* I had been executing big efforts ad-hoc and keeping knowledge only in chat/memory instead of routing it to these durable, shared homes.

**How to apply (my improvement — the coherence rule):** treat the instruments as ONE BODY of project knowledge with a clean separation of concerns — operational→board, decisions→ADR, visions→registry, benchmarks→reference-platforms, strategy→plan/, findings→audit/, durable know-how→knowledge/patterns/, cross-session→memory. For any non-trivial effort: (1) name the journey (`11`); (2) PLAN it into the right instrument(s) — board cards always; an ADR/plan-doc/registry entry when the situation earns it (design rationale, alternatives, a vision, a benchmark) — judged per situation, neither always nor never; (3) EXECUTE the plan faithfully in order — deviation → pause + re-plan on the instrument, never silent drift; (4) as work lands, UPDATE the instruments (close cards, write the ADR, register the vision, log the audit finding) so nothing is scattered and nothing is lost. The judgment of WHEN to reach for WHICH is mine and must be right — situation-fit, senior-leader, never rigid doctrine.
