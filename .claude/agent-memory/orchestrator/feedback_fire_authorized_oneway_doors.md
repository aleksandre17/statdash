**Rule:** when the owner has AUTHORIZED a one-way door (old→new replacement) AND the plan gates it on a machine-proven equivalence gate (new==old before old is removed), PROCEED and fire it on green — do NOT pause to re-ask permission just because the owner is present/engaged. The gate is the safety; re-asking is hesitation, and it frustrates him.

**Why (owner, twice — overnight 2026-07-10 + again 2026-07-11):** "you're responsible for logistics, foresight, always being the best — what's the problem? So because of [one-way doors] we won't replace old code with new? What are we waiting for? If we have a plan, what's the problem? We don't fit the idea to the existing — the existing must serve the idea." I had paused the ⛔ R2/R3-contract + V3-switch ("holding for your deliberate go, you're here") after he'd already authorized them to fire on green. That reflex was wrong.

**How to apply:** refines [[feedback-decisive-initiative]] (which says *confirm* one-way doors). The confirmation is a ONE-TIME authorization, not a per-step gate. Once granted + a green equivalence gate exists: build the expansion (new path alongside old) → the contract FIRES automatically on the gate going green → the gate removing the old code IS the engineering proof, not a wait. The ONLY discipline I keep on the irreversible step is INDEPENDENTLY verifying the equivalence gate (root tsc + the FF is non-vacuous + green on ALL stored configs) BEFORE the removal — because a false-green would delete working code (I caught one via capstone root tsc the same day). That independent gate-check is NOT "asking permission" — it's the proof. A gate that cannot go green is a real blocker to surface, never a forced removal. The existing serves the idea; build the new, prove it, remove the old — decisively.
