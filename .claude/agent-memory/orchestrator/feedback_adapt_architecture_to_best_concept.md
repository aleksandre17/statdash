When fixing or improving ANYTHING, anywhere on this platform: find the canonical, best-in-class concept FIRST, then reshape the existing architecture to it. NEVER bend the best concept down to fit the current pattern. A locally-clean fix that preserves a mediocre pattern is REJECTED — that is the ceiling the user holds.

**Why:** The user repeatedly rejected fixes that merely RELOCATED a smell instead of reconceiving it. Concrete case: the `SiteRenderer` magic-key violation `vars['_pageColor']` → the proposed fix was `presentation['color']` — still the generic renderer imperatively special-casing each concern (`typeof` checks, casts, a per-concern branch), just reading from a different bag. The user's exact words: *"I'm not telling you to adapt the architecture to a found solution — I'm telling you the existing [architecture] must adapt to the highest solution and concepts. Always the best; the existing bends to the best, never the reverse. This applies to everything, everywhere."* This is also why they keep asking for independent senior/scientist/architect thinking — they want the conceptual CEILING, not a passing patch.

**How to apply:**
- Frame every design/agent task at the CONCEPTUAL level: *"what is the canonical, best-in-class way the best platforms/standards model this?"* — then migrate our code to THAT. NEVER anchor a task to "rename/relocate/typed-ify the current pattern" — that anchoring is exactly what produced the rejected fixes.
- Shared/generic layers (renderers, engines, registries) must be CLOSED for modification, OPEN for extension: a new concern = a new registered capability/projector, the shared body UNTOUCHED (OCP / Law 8 M-5). If a fix makes a shared layer grow an `if thisConcern / if thatConcern` branch or special-case a named key, it is wrong — reach for the polymorphic/registry/declarative model the codebase ALREADY uses elsewhere (open palette, store-builder registry, command bus, `FieldControlRegistry`).
- Don't settle for "good enough / locally clean." Reinforces Laws 4 (full standards), 6 (best solution), 7 (architecture leads), 8 (platform-level/OCP) — the user's emphasis is the highest conceptual bar across the WHOLE finish-line effort, not one file.
- When routing to agents, pass THIS bar explicitly and name the failure mode (relocating ≠ reconceiving), or they default to a local fix. Pairs with [[verify-board-empirically]].

**Orchestration mandate (added after a repeat miss):** a MECHANICAL task framing produces MECHANICAL relocation. Every dispatch prompt MUST mandate conceptual depth — "find the canonical best concept, eliminate the ROOT-CAUSE hardcode (not move it), anticipate future needs (foresight), reveal senior/architect vision" — NOT "decompose / refactor for SRP / do this specific change." Example of the failure: a "decompose SectionShell for SRP" task correctly split it into hooks/sub-components but LEFT the real hardcodes — magic BEM class strings (`'section--hero'`) and inline variant→class logic (`view.hero && 'section--hero'`) — because the prompt asked for relocation, not reconception. The user (rightly): "this is just relocation, not architectural/conceptual improvement; everyone completes the literal task and beyond that nobody reveals creative/critical/independent thinking, foresight, always-best." The "best" must be VISIBLE in the output. Hold YAGNI too — depth ≠ gold-plating.

**Genericity / shared-placement reflex (the user caught this 3+ times):** when ANY helper, hook, type, or module is created/extracted, the MANDATORY question is "is this generic — reusable by OTHER elements, now or future?" If yes, it belongs in the SHARED layer (`packages/react`, `packages/styles`, `packages/core/contracts`), NOT trapped in one element's folder (`packages/plugins/nodes/<element>/`) or named after one element. Concrete misses the user caught: `useCollapsible` (a pure disclosure/ARIA hook) + `useViewToggle` (role-based view switching) + `sectionAccentStyle` extracted into `nodes/section/default/` instead of shared — trapping generic behavior + privileging section; `core/config/section.ts` named for one element but holding the platform-wide config vocabulary (`DataSpec`/`VisibilityExpr`/`resolveTemplate`); `navUtils` hardcoding `section`/`georgraph`/`row`. EVERY extraction/refactor prompt must mandate this genericity test + correct placement, and a privileged/element-named module holding generic vocabulary is a smell to split + de-privilege. Pairs with the no-privileged-element / capability-registry discipline.
