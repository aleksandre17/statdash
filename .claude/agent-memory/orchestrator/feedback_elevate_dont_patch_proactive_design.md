When the work is UI / responsive / design, "fix the breaks" is the FLOOR, not the goal. The owner wants the whole design ELEVATED to the highest market standard, coherently, proactively.

**Why (the owner's words, 2026-06-28):** after a responsive pass that fixed real defects, the owner said it "is NOT the best responsive design" — it was reactive CSS-patching, not elevation. Three specific failures: (1) it over-corrected the page measure (1280px hard cap → inner pages feel squeezed; a data dashboard needs to breathe), (2) it MISSED an obvious quality issue — the KPI strip squeeze (fixed `repeat(4/2,1fr)` grid instead of intrinsic `auto-fit minmax`) — the owner *deliberately waited* to see if we'd catch it, and we didn't, (3) it ignored the platform's own capabilities. "I asked you for individualism, vision, professionalism, observation — why do you struggle and need instructions? I want senior scientist/engineer/architect/developer, pattern/concept/architecture/platform-based standards."

**How to apply:**
1. **Elevate, don't patch.** Charge design agents to evaluate the WHOLE UI as a senior product-designer + architect and raise it to best-in-class (Tableau/PowerBI/Looker/Stripe-dashboard-grade), not to clear a defect list.
2. **Use the platform's own layout system.** `packages/plugins/nodes/layout/` has `grid · columns · stack · row · wrap · card · spacer · divider` (responsive node primitives). Prefer composing/arranging with these (responsive props) over ad-hoc CSS. The platform should eat its own dog food.
3. **One common guideline.** Every element's styling obeys a single design-language spine (measure, spacing scale, intrinsic responsive grids `auto-fit/minmax`, container-queries, breakpoint tokens) grounded in the highest market standard. The canonical doc is `platform/work/DESIGN-css-responsive-standard.md`.
4. **Be proactive — observe.** Find the cramp/squeeze/imbalance/illegibility WITHOUT being told (real-browser, full ladder). The kpi-strip miss must not recur. Give agents a VISION charge, not a checklist.
5. **Intrinsic > breakpoint.** Fixed column counts and hard max-width caps are the smell; prefer fluid `clamp()`, `auto-fit minmax` grids, and `@container`. See [[feedback_css_architecture]] and [[localestring-leak-apex-blindspot]] (real-browser verify before claiming done).
6. **The 3-tier style-override cascade (owner's canonical model, 2026-07-01).** ONE consistent mechanism on EVERY node: (1) plugin element ships IDEAL responsive styles by default (full `@container`/breakpoint coverage — perfect with zero config) → (2) our JSON `config`+`styles` override it (responsive-aware) → (3) any INNER node can override the OUTER node's style, again via JSON config+style (specificity: inner wins). Getting this cascade clean + uniform "keeps becoming the challenge" — it's a design-architecture problem, not a patch. Canonical home: `platform/work/DESIGN-responsive-composition.md`.
7. **Compose via layout nodes — NO bespoke section/page divs.** The owner rejects hand-authored "two divs, one holding 2 sections + one holding 1" when a configurable layout node (`packages/plugins/nodes/layout` columns/grid/stack…) renders that DYNAMICALLY-responsive (breakpoint column-collapse, authorable `align`/count/gap) from JSON. Section/page STRUCTURE must be layout-node composition, ONE consistent "handwriting", bespoke divs retired (Strangler). This is maximal-adoption of our own primitive — [[maximal-adoption-doctrine]].

**RECURRENCE LESSON (2026-07-01):** this exact feedback was already in memory from 2026-06-28, yet the styles lane was briefed NARROWLY again (fix panel-sizing + R1/R2/R3) and missed the vision — the owner had to re-raise it ("why doesn't your best agent think with broad vision?"). The failure was the ORCHESTRATOR under-briefing, not the agent under-thinking. **Operationalize: the lead must ACTIVELY inject this VISION into EVERY UI/responsive/composition brief (the cascade + layout-node adoption + elevate-not-patch), not passively hold it in memory.** Infer the broad intent; expand scope; never ship a narrow fix where the owner wants an elevation. See [[orchestrator-briefing-doctrine]].

**HARD CONSTRAINTS (never, on any change — the owner restated 2026-06-28):** vision/individualism/criticality must SERVE improvement, the highest-standard research+implementation, and broad-perspective thinking — and in NO case produce regression, hardcoding, over-rigid "blueprinting", antipatterns, static-ization (killing responsiveness/adaptivity), worsening, or any degradation. The orchestrator ENFORCES this: every returned change passes the full converged green-gate AND real-browser verification across the ladder before it ships; anything that degrades an existing guarantee is rejected/reverted (the rollback net is real), not relayed onward. This is guardian duty — see [[feedback_guardian_of_canon]], [[feedback_resolve_dont_defer]], [[feedback_maximal_adoption_doctrine]].
