## The durable heuristic (why the fixture couldn't catch it)
**A fixture that models a "total" as the SDMX literal `'_T'` cannot exercise a total carried at a CONCRETE code** (a country code `GE`, an "all"/`_Z` approach, a `TOTAL` sector). The real cube encodes "whole-country/whole-X" as a concrete member that ALSO functions as the aggregate. Any predicate that special-cases `'_T'` (`isConcreteMember`, total-detection, rollup exclusion) is blind to this. When a browse/aggregation "renders in the `_T` fixture but reads empty live," suspect the live total is a concrete code and the `_T` special-case is the blind spot. Ground-truth the ACTUAL member values from the facts bundle / `DATA/canonical/*.json` (the cube's `nonTimeDims` + real dim_key), never the fixture's shape. Sibling of [[metric_id_filter_selfmatch]] (fixture self-matches what the runtime resolves differently) and the [[async-store-live-render-patterns]] two-store contract. Repro-first paid off here: theory kept predicting "should render rows"; a faithful single-`ExternalStore` repro with the live obs shape (`measure=slug, approach='_Z', geo='GE'`) showed the null-values reality in one run and bisected to the kept geo pin.
