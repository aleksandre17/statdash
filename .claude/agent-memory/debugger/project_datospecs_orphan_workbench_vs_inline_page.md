Two distinct DataSpec homes on the statdash platform — do not conflate them when a "spec is corrupted" report comes in:

1. **Published/draft page surfaces embed their DataSpec INLINE** inside `page.config` (each section element carries `data: {type:"query"|"pipeline", query, pipe, encoding, ...}`). Pages do NOT reference standalone spec rows by id — a grep of every page (published + draft) for the workbench spec UUIDs returned zero hits. The page's own `data_specs[]` array is empty. Baseline for these lives in git: `platform/apps/api/provisioning/geostat.provisioning.json` (`pages[]`), and the live inline spec can be diffed byte-for-byte against it.

2. **`GET /api/config/data-specs` is the authoring workbench's "suggested"/scratch collection** — auto-generated by `platform/apps/panel/src/discovery/suggestPanels.ts` (profile → ranked panel types) and named `"<panelType> (შემოთავაზებული)"` ["(suggested)"]. These are NOT in provisioning (`provisioning.json` top keys are only `version/pages/siteConfig/dataSources` — no data-specs) and NOT in git. They back NO rendered surface. There is therefore **no clean baseline** for them anywhere.

**Incident (2026-07-20, dev :3013, API 192.168.1.199:3011):** owner reported "the GDP dynamics part's pipeline is messed up" after experimental auto-saved edits — framed as a sibling of the regional restore. HYPOTHESIS (from prior agent): a corrupted GDP-dynamics *spec* backing a live surface, candidate `d954e1ba` (timeseries-suggested, source metrics `[accounts.gdp, gdp.perCapitaGrowthYoy, probe_regional_gva, probe_gva_mrq5c8ox]`, `encoding.value="GDP_AGRI"`). **FALSIFIED.** The published GDP-dynamics surface (gdp page section id=`gdp-dynamics`, "მთლიანი შიდა პროდუქტის წლიური დინამიკა") is pristine: inline spec `measure:"gdp.current"`, filter `{geo:GE, approach:_Z}` — byte-identical to provisioning, resolves 16 rows (2010: 22,148 → 2025: 104,598 GEL mn via raw measure `gross-domestic-product-at-current-prices`). Page untouched since 09:26; the corrupt edits landed 10:01–13:01. The corruption is confined to ~8 orphan workbench "suggested" specs whose `encoding.value`/measure was flipped to `GDP_AGRI` (a REGIONAL sector code, resolves 0 rows against the GDP cube) — all orphan, no baseline.

**Why:** Unlike the regional incident (a data-source `datasetCode` flip that corrupted a *published* surface through the store-routing layer), the GDP data-source was already verified good, so no published surface was ever broken. The "spec corruption" was real but only in unbound authoring scratch.

**How to apply:** When a "spec is corrupted / dynamics broken" report arrives — first locate the surface the user actually sees (inline page config, diff vs provisioning) BEFORE touching the `/api/config/data-specs` collection. If the only broken copies are orphan workbench "suggested" specs, there is no clean restore target (do not fabricate one — provisioning/git have none, and the session BACKUP-live-data-specs snapshot is taken mid-incident so it is POST-corruption). Recommend deletion as hygiene, but confirm before removing the owner's in-flight authoring artifacts. Sibling notes: [[config_source_versioning_gap]], [[preview_path_store_routing_ssot]].
