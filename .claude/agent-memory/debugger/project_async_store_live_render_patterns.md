## The meta-pattern
The platform runs TWO store implementations behind one `DataStore` port: a **sync in-memory `ExternalStore`** (static kind) and an **async `ApiStore` (+`CachedStore` wrapper)** (stats/live kind). Any per-spec scope (measure pin, filter, `$ne`, range) honored by one path but dropped/defeated by the other is **invisible until live** — green unit tests pass because the async contract is tested where it isn't used and used where it isn't tested, and the render probe only checks empty/errors so it passes while VALUES are silently wrong.

**Diagnosis playbook (how these were actually caught):**
- Dump the OBS WIRE requests (`?dataset=…&from=…&to=…&filter=…`). Tells: a missing unbounded read; a wrong from/to; a `filter` missing `measure` on a val query; two panels firing the identical filterless query; a stale ctx pin defeating a `$ne`; `filter={"geo":"R2,R3"}` sent as one literal.
- READ SCREENSHOTS with real data — value-correctness and color defects render with 0 console errors and slip past build/typecheck/unit green. Metric-green ≠ correct.
- The OLD static app's bundled config (even minified defaults like `is=String(ns[ns.length-1])`=max year) is the **parity oracle for INTENT** — resolve its vars; never self-reference the current manifest.
- When a bug reproduces for the owner but never for probes, suspect a hard-vs-soft-nav divergence — see [[probe-methodology-hard-vs-soft]].

## The trap catalog (each was a real, fixed bug)
1. **Per-spec pin dropped on the async store.** `storeVal {type:'val',code}` — `ApiStore.toObsParams` didn't map `q.code` to the wire `filter`, so every val read returned ALL measures and took `rows[0]` (all three `_Z` KPIs showed the same number). Fix: pin `filter[MEASURE_DIM]=q.code` for val queries; `MEASURE_DIM='measure'` SSOT in `core/context.ts` (mirrors `TIME_DIM`). Guard: `kpi-value-binding.fitness.test.ts` proves pin EFFECT, not just existence.
2. **Cache identity must reflect EVERY field that changes resolved data.** `extractRequirements('query')` ignored `spec.query.filter` → two approach-pinned panels (PROD vs EXP) got identical `specDimKey` → module-level `_promiseCache` served the first's rows to the second (Expenditure panel rendered Production's values). Separately, the async `_promiseCache` in `useNodeRows` keyed only on `depKey`(fetch fingerprint), NOT node-unique → sibling nodes with the same covering fetch (geo-map + sector pivot) collided → cross-filter State-B showed "GDP by region" not the sector breakdown. **Fix: `cacheKey = recipeKey ⊕ depKey`** — recipeKey = stableStringify(`node.data` + `node.transforms`) = per-NODE axis; depKey = per-STATE axis. It hid because the SYNC fast-lane calls `resolveNodeRows` per instance (per-instance memo, no shared cache), so sync parity harnesses never reproduced it. Also fold `ctx.pageStoreKey` (multi-store routing) into the key — still uncovered as of the State-B fix (chief-engineer systemic #1). `useKpiRows.kpiDepKey` includes `specs` on its static branch but the hot reqs branch is code×dims with NO recipe — same latent class.
3. **Warm-key ≡ read-key.** `useKpiRows` built the warm ctx as `{...sectionCtx.dims, ...r.dims}` — the spread REINTRODUCED a dim that a KPI filter wildcards (national-total KPI on a region-pinned page: `withFilter` drops `geo`, spread put it back) → warm key had geo=R2, read had none → cache miss → `ApiStore.querySync called cold` → NodeErrorBoundary. Fix: warm under `r.dims` VERBATIM (it IS the authoritative resolved read ctx).
4. **Wire route can't express `<>` or multi-literal — resolve client-side.** The observations route expresses ONLY `@>` containment + `= ANY` (array); its Zod schema REJECTS object filter values. So `$ne` and comma-multi are CLIENT-SIDE-only. `{$ne}` fell to `String(fv)`→`"[object Object]"` (empty map); a comma `$ctx` (`geo:"R2,R3"`) was `String()`'d as one literal → 0 rows. Fixes in `store-filter.ts`/`store-api.ts`: drop `$ne` from the wire (send only a co-located `$ctx` positive scope) and exclude post-fetch via the SSOT `matchesFilter`; split a comma `$ctx` into the array the array-branch already emits; **fold the resolved `$ne`/multi into the cache key** so `{$ne:'_T'}` and `{$ne:'R2'}` over the same wire slice don't collide. The in-memory store worked because its `resolveFilter`/`matchesFilter` split comma + evaluated `$ne` in memory — make the wire serializer and the client matcher AGREE (SSOT).
5. **A pure `{$ne}` must DELETE the ctx baseline pin.** `toObsParams` pre-seeds `filterRecord[dim]=ctx.dims[dim]` for every nonTimeDim first; a pure `{$ne}` (no `$ctx`) did nothing → the stale `sector:'_T'` stayed on the wire → fetch scoped to `_T`, then client `$ne:'_T'` dropped every row ("No data" donut). Fix: in the pure-`$ne` branch `delete filterRecord[dim]` so q.filter intent wins over the ctx baseline (fetch broad, exclude client-side).
6. **Auxiliary classifier not loaded.** A `{$cl}`/`{$d}` join injects nothing when the joined classifier is AUXILIARY (keyed by measure code, e.g. `aggregates` carrying isClosing) not a wire-filter DIMENSION. Store-builder loaded classifiers only from `nonTimeDims`; fix = load from `classifierDims` (superset), union nonTimeDims; ApiStore still gets nonTimeDims for the wire. Also the classifier ACL dropped `metadata` — `liftClassifierMetadata` (generic, Law 1) lifts every scalar metadata key to a first-class entry attr.
7. **Vite tree-shakes `register*()` side-effect modules.** A declarative pipe/registry that "does nothing" LIVE but works in a Node repro off built dist = SUSPECT TREE-SHAKING. `registerTransformStep(...)` calls live as bare module-init in `data/transform/index.ts`; that file wasn't in core's `package.json` `sideEffects` allowlist → geostat's Vite (resolves `@statdash/engine` to SOURCE) tree-shook them → `listTransformOps()`=[] at runtime → every pipe step a silent NO-OP (axis derived from raw value, but no `label`/`series`/`_id` → "axis but no bars"). Fix: add the file to `sideEffects`. Check the allowlist covers EVERY file with bare `register*(...)`.
8. **Dataset-level ETag 304s uncached slices to empty.** `ApiStore.queryAsync` sent `If-None-Match` whenever a dataset ETag was known, but the ETag is DATASET-level while each `(from,to,filter)` slice is its own `_cache` entry → after the first slice 200s, later slices 304 and the branch returned `cache.get(key) ?? []` WITHOUT caching → post-warm `querySync` cold-throws (kpi-strip crash). Fix: gate on a per-slice cache hit `if (storedETag && this._cache.has(cacheKey))`. (NB chief-engineer systemic #2: this over-corrected into dead 304 code — verify.)
9. **Order-unstable cache key.** `toObsParams` assembled the filter record in source-dependent order → same logical slice serialized two ways → warm/read key divergence. Fix: sort filter keys before `JSON.stringify` (SSOT cache identity + deterministic wire URL).
10. **Async default: distinguish loading(null) from empty([]).** `from=0&to=0` HTTP-400: `getOptions`→`resolveYears` returns `[]` (not null) when the `time` classifier isn't loaded yet → `resolveDefaults` doesn't push to pendingKeys → `isLoading=false`, render not gated → `autoParse('year-select','')` = `Number('')||Number(objDefault)||0` = literal `0` → survives the `!== ''` filter → `time=0` → wire `from=0`. Two seams: async-empty-vs-loading hole + sentinel coercion. When debugging "0 where a value should be" in filter defaults, check whether the options source is async and whether getOptions returns null (loading) vs [] (empty).
11. **CachedStore masks source caps.** `CachedStore` hardcoded `caps.sync=true` regardless of source and implemented no `queryAsync` → wrapping an `ApiStore` (caps.sync=false) reported sync=true → `resolveStore`'s async bypass never fired → sync `interpretSpec`→`querySync` hit the cold ApiStore → threw → empty charts ("endpoints have data but UI empty"). Correct fix = CachedStore inherits source caps + implements queryAsync AND renderNode routes caps.sync===false through the async `useNodeRows` hook — NOT eager whole-cube prefetch or catching the throw.
13. **Module async cache omits the store INSTANCE → stale/poisoned across a CANVAS store rebuild.** Closes #2's residual. `useNodeRows`/`useKpiRows` `_promiseCache` key = recipe/req ⊕ data-state ⊕ store-KEY — complete on the runner (map built once) but the canvas `useLivePreviewStores` REBUILDS the map on any source edit → new COLD `ApiStore` gen under the SAME key; a node warmed on the prior gen REUSES its promise (same-cube ⇒ same value ⇒ invisible, why KPIs "looked fine"; transiently-errored ⇒ poisoned till reload, Retry re-reads the rejection). depKey-moved nodes re-warm → "some switch, some don't". Fix: shared `storeGenId(store)` WeakMap (`resolveNodeRows`) folds a per-INSTANCE id into both keys (stable store ⇒ byte-identical). Guard `storeGeneration.fitness.test.tsx`. Eviction-on-reject STORMS (settle-evict + Suspense auto-retry loops forever) — recover by cache MISS only; a working single-gen Retry needs a NodeErrorBoundary↔cache seam (OPEN). Map tail: GeoMap fetches geojson via Worker from root-relative `def.geoJsonUrl`; panel origin not serving `/data/*.geojson` ⇒ blank map regardless of data (verify the asset).
14. **Mode effect clears a dim a $ctx ref needs.** A CURRENT-mode KPI reading `from:{$ctx:'fromYear'}` where fromYear is a RANGE-ONLY bar dim that the `mode!==range` effect sets '' → resolves to '' → cagr guard false → 0.0%. For "over the whole series" metrics, drive from a **span-derived HIDDEN param** (min/max year from the cube via `{from:'options',pick}`), present in a bar that's visible in every mode, NOT the user's range selectors (hardcoding literal years is a symptom patch — breaks on data update). Filter DEFAULTS must respect bar visibility (`evalWhen(bar.showWhen)`): don't cross-pin a hidden mode's param. Span dims are safe as ctx.dims because `buildObsFilterParam` only pins dims in `nonTimeDims`.

## Render / CSS correctness traps (same "0 errors but wrong" class)
- **`??` vs `||` on lookup/display values.** A `{$d}` lookup yields `''` (empty STRING) for an unmapped key; `??` KEEPS '' (transparent fill / unset pin) — use `||` for color/string fallbacks. Bit treemap tiles (invisible), unmapped colors.
- **Value-shaded encodings** (choropleth/heat): use token-derived `sequentialRamp()`+`quantileColors()` from `@statdash/styles` (rebrands under `[data-tenant]`; quantile-by-rank so a dominant outlier still spans the ramp), never a hardcoded palette or flat single fill. The `panels/map` `buildColorScale` is a legacy token-blind stub — don't conflate it with the real `geograph` node.
- **Categorical SVG charts** (donut/treemap/custom) with no per-datum color: distribute `chartPalette()` (mirror the treemap's `distribute = distinct.size<=1` guard) — don't fall to a single muted grey.
- **Custom CSS-flex charts** (`height:100%`) need EVERY ancestor up to the height-constrained box to carry height — an unstyled a11y/semantic wrapper (`<div aria-hidden>`) breaks the chain; Apex masks this by self-measuring. A `.panel` node needs its own height flex-chain — `aspect-ratio` on a flex `.panel-col` computes `auto` (no-op). (Height model since unified — see architect chrome/panel/section unification.)
- **Map render.** GeoMap is DOM-free d3-geo SVG now (Leaflet hidden-container `d="M0 0"` trap retired). New risk → trap #13 tail (geojson Worker asset).
