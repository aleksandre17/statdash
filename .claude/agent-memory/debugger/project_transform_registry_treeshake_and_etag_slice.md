The accounts hbar-diverging charts drew an axis but NO bars/labels, and GDP/REGIONAL/ACCOUNTS dynamics-mode kpi-strip crashed (`ApiStore.querySync called cold`). Both were LIVE-ONLY (green tests, fine in-repro) — invisible until read on the deployed geostat. Three root causes, all in `platform/packages/core`:

**Bug 1 — transform-step registry tree-shaken to EMPTY (accounts bars).** The config pipe (`join`/`lookup`/`sort`/`group`/`template`/`concat` with `{$cl}`/`{$d}`) ran but every step was a SILENT NO-OP, so raw obs rows reached the chart with no `label`/`series`/`_id`/`_isGroup` → empty categories → no bars (the value axis still derived from raw `value`, hence "axis but no bars"). Cause: `applyStep` does `getTransformStep(op) ?? rows` (unknown op → passthrough, NO throw — `transform/pipeline.ts:14`). The 18 `registerTransformStep(...)` calls live as bare module-init statements in `data/transform/index.ts`, but that file was NOT in core's `package.json` `sideEffects` allowlist. The geostat Vite build resolves `@statdash/engine` to SOURCE (`exports["."].source = ./src/index.ts`) and tree-shakes any non-whitelisted module's side-effect statements → registry empty at runtime (`listTransformOps()` returned `[]` live). Fix: add `"./src/data/transform/index.ts"` to `sideEffects`. (Charts whitelists `./src/interpreters.ts` for the same reason — that's why the chart INTERPRETER worked but the engine PIPE didn't.)

**Bug 2 — dataset ETag 304s an uncached slice to empty (dynamics kpi-strip crash).** `ApiStore.queryAsync` sent `If-None-Match` whenever a dataset ETag was known, but the ETag is DATASET-level while each `(from,to,filter)` slice is its OWN `_cache` entry. After the first slice 200s + stores the ETag, every LATER slice of that dataset 304s — and the 304 branch returns `this._cache.get(cacheKey) ?? []` WITHOUT caching, so that slice stays uncached. The post-warm `querySync({type:'val'},thatSlice)` then cold-throws (kpi-strip shell crash). The kpi warm (`useKpiRows`) fires many per-year val reads (yoy t-1, cagr from/to) → first 200s, rest 304-to-empty. Fix: gate the conditional on a per-slice cache hit — `if (storedETag && this._cache.has(cacheKey))`. Regression test `apiStore.async.test.ts` (4 + 4b); the OLD test 4 had ENCODED the bug (expected If-None-Match on a different/uncached slice) — updated to the correct contract.

**Bug 3 (hardening) — order-unstable cache key.** `toObsParams` assembled `filterRecord` in a source-dependent order (nonTimeDims loop, then the val measure pin, then q.filter) → the same logical slice could serialize `{approach,geo,measure}` vs `{approach,measure,geo}` → two cache keys for one slice → warm/read key divergence. Fix: sort filter keys before `JSON.stringify` (SSOT for cache identity + deterministic wire URL).

**Why this class is dangerous / how to apply:** the render probe only flags empty/errors, so a no-op pipe (axis derives, bars vanish) passes silently — caught only by reading screenshots + dumping the interpreted row shape (`sampleKeys` lacking `label`/`series`/`_id` = pipe didn't run). When a declarative pipe/registry "does nothing" live but works in a Node repro off the built dist, SUSPECT TREE-SHAKING of side-effect module-init: check the package's `sideEffects` allowlist covers every file with bare `register*(...)` calls. For async-store warm/read cold-throws, the cause is a warm slice that never landed in cache — check (a) per-slice vs dataset-level revalidation (304-to-empty), and (b) order-stable cache keys. Mirrors [[apistore-val-measure-drop]] (two stores must satisfy one contract) and the warm/read-key SSOT. Verified by deploy→screenshot: NEW accounts reached byte parity with the OLD app (:5171) — 261 CSS bars, 0 No-data, dynamics crash-free on all 3 pages.
