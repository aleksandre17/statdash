`LocaleString = {ka,en}` (bilingual display labels) flow end-to-end as tagged objects and are resolved to a single locale ONLY at the React render boundary (`resolveNodeRows`/`resolveRowLocales` for tagged row cells; `useResolveLocale`/`useResolveLocaleSafe` in shells). The i18n boundary is **POSITIONAL**: any place that `String()`s or COMBINES a cell into text must either (a) be at/after the React resolve boundary, or (b) preserve + re-tag the LocaleString. Never let the locale-agnostic engine pick a locale (Law 1), and never flatten before React. Symptom of a violation: literal `"[object Object]"` on the live site, or React error #31 ("Objects are not valid as a React child, found object with keys {ka,en}").

## Three facets (each was a real, fixed bug)

**1. Discriminate objects by FIELD IDENTITY, not shape.** `resolveRowLocales`/`isLocaleObject` discriminated a LocaleString by STRUCTURE (`typeof==='object' && !null && !Array`). But `LocaleString`, `ProvenanceRecord` (`DataRow.provenance`), and `seriesFormat` (`Record<string,string>`) are ALL plain objects — structurally indistinguishable. So the resolver flattened `provenance` to `Object.values(v)[0]` (a random scalar) → `r.provenance?.status` became undefined → preliminary/last-updated/methodology badges silently stopped (Law-9 regression). Fix: skip a known `NON_LOCALE_ROW_FIELDS = {provenance, seriesFormat}` denylist — a genuine display LocaleString only ever enters via a `$d` join under a display-attr key, never a reserved key. (Chief-engineer systemic #3 flags the residual weakness: this is a DENYLIST, not a positive tag — the next structured non-LocaleString field is flattened unless added. Prefer a positive brand/tag.) Principle: "make illegal states unrepresentable" — when two object-typed values can't be told apart by shape, discriminate by a known key/tag.

**2. Combine PER-LOCALE and re-tag (composeLocale), don't flatten in the pipeline.** `applyTemplate`/`applyConcat` (`core/data/transform/steps.ts`) did `String(row[f])` for `{label} ({measure})` → baked `"[object Object]"` into the label DURING the pipe, BEFORE React, AND destroyed the tag so resolveRowLocales had nothing to resolve (worst leak: accounts table `<th>` = 318 hits). `resolveOptions`/`resolveChips` did `String(o[labelField])` → `<option>[object Object]`. Fix: `composeLocale(operands, render)` + `localeKeysOf` in `core/i18n/types.ts` — when any operand is a tagged LocaleString, compose the result PER LOCALE and re-tag (`{en:'GDP (B1)', ka:'მშპ (B1)'}`) so it flows through resolveRowLocales like any `$d` label; all-scalar operands → plain string (byte-identical). `SelectOption.label`/`ChipOption.label` retyped `LocaleString`; shells resolve at the boundary via `useResolveLocaleSafe()`. FLAGGED (same class, lower priority): `sort`/comparison ops still `String(LocaleString)` for `by:'label'` → compares all as "[object Object]" (no real ordering; sort on a resolved/scalar key).

**3. An honest TYPE is what lets the compiler catch a missing resolve — and the fitness matrix must cover EVERY page.** The landing/portal React-#31 crash: `StatsCarouselShell` rendered `{stat.unit}` RAW because `StatItem.unit` was typed `unit: string`, but bilingualization made it a `{ka,en}` in config — so TS never flagged the missing resolve. Two-part fix: make the type honest (`unit: LocaleString`) AND resolve at the boundary (`{resolve(stat.unit)}`, byte-identical for plain strings via string-passthrough). The GAP that let it ship: the render-guard fitness matrix's PAGES list was `['gdp','accounts','regional']` — it OMITTED the index/`landing` page. Any page-render fitness gate must DERIVE/cover the FULL page set (index/portal included), not a hand-listed subset, or a whole page composition renders untested.

## How to apply
When adding a new pipeline op or a new label-bearing render surface, route it through `composeLocale` (combine) or `useResolveLocaleSafe` (render). When a config-sourced display field is bilingual, make its shell type honest (`LocaleString`, not `string`) so the compiler enforces the resolve. Keep page-render fitness matrices FULL-page-set derived. NOTE: `packages/react` is app-agnostic — a fitness test blocks tenant strings even in fixtures; use a neutral name like 'StatsOffice'/'StatsOffice', not 'Geostat', and neutral locales `'en'/'fr'` in engine/react tests.
