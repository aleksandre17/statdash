The last two geostat render residuals after the dynamics/geomap/sector parity pass, both LIVE-ONLY (1068 core+react+plugins+charts tests green). Verified by deploy→screenshot vs OLD (:5171). NEW data vintage differs from OLD: REGIONAL spans 2010–2023 (2010=21821.57, 2023=80882.76 → CAGR 10.6%); OLD spans 2010–2024 (→10.7%). GDP/ACCOUNTS approach breakdowns exist 2010–2024 only; 2025 has only the `_Z` headline measures.

**RESIDUAL A — regional CURRENT-mode CAGR KPI showed 0.0% (should be 10.6%).** The `reg-cagr-year` KPI (`apps/api/provisioning/geostat.provisioning.json`, mode:'year') read `from:{$ctx:'fromYear'}` / `to:{$ctx:'toYear'}` — but fromYear/toYear are RANGE-ONLY ctx dims (live in the range-bar, `showWhen mode==='range'`) and the `mode!==range` effect SETS them ''. In year mode they're absent → engine `resolveTime` (kpi.ts:86 `ctx.dims[ref.$ctx]`) → '' → cagr guard `vFrom && to>from` false → 0. This is the inverse of the dynamics GAP A: a CURRENT-mode KPI depending on range-mode dims that the mode effect clears, AND bar-visibility gating ([[dynamics-geomap-sector-parity]] GAP A) denies them a default in year mode.
- **Fix = a new Constructor-ready capability: span-derived HIDDEN state.** A headline "average annual growth over the WHOLE series" is span-driven, NOT selector-driven — and the OLD computes the span from data (label auto-fills "2010–2023" / OLD "2010–2024"), so HARDCODING literal years is a symptom patch (breaks when data updates). Added `options?: OptionsSource` to `ParamHidden` (`packages/core/src/config/filter-params.ts`) — "a hidden state var whose Tier-3 OptionsDefault `{from:'options',pick}` follows the cube". Taught `useFilterState.getOptions` (`packages/react/src/filters/useFilterState.ts`) to resolve a hidden param's options like a select (else it returns null → stuck pending → isLoading forever). Then added `spanFrom`/`spanTo` hidden params (asc-first=min, desc-first=max year) in BOTH the year-bar AND range-bar (one bar always visible → resolves in every mode), NOT cleared by mode effects, mapped in context.dims, and repointed `reg-cagr-year` from/to + label `{spanFrom}–{spanTo}`. Regression: `useFilterState.spanDefault.test.tsx`.
- SAFE that spanFrom/spanTo are ctx.dims: `buildObsFilterParam` only pins dims in `nonTimeDims` (the cube-dimension list from the datasource config), NOT every ctx.dim — so span* never reach the wire filter (mirrors the discipline in [[apistore-val-measure-drop]]). `atTime` only sets `time`; span* are read-only refs.

**RESIDUAL B — GDP CURRENT income TREEMAP rendered BLANK ("Prelim." + empty box).** TWO stacked bugs, both in the chart render path (pre-existing, NOT caused by A). The data was FINE (the table tab showed all 4 income components):
1. **Height-chain collapse.** `Chart.tsx` wraps the chart renderer in `<div aria-hidden="true">` with NO height. The custom flexbox `TreemapChart` root is `height:100%` → resolved against a content-sized wrapper → tiles collapse to 0. (Donut/bar are apex and measure their own box, so they survived — only the CSS-flex treemap collapsed; same family as the regional GeoMap height-chain in [[dynamics-geomap-sector-parity]] GAP B.) Fix: gave the wrapper `className="chart-wrap__render"` with `height:100%` (and `height:auto` ≤1280px, matching `.chart-wrap`) in `packages/plugins/panels/chart/default/chart.css`.
2. **Empty-string color kept by `??`.** With size restored the tiles were still INVISIBLE: `background: item.color` was '' (transparent). The `{$d:'measure'}` lookup yields color '' (empty STRING) for an unmapped measure; `pt.thresholdColor ?? base` and `base = series[0]?.color ?? cssVar(...)` and the interpreter's `chartRows[0]?.color ?? DEFAULT_ACCENT_COLOR` all KEEP '' (nullish `??` only catches null/undefined). Fix: `??`→`||` at all three sites (`TreemapChart.tsx` ×2, `charts/src/interpreters/special.ts` TreemapInterpreter). All tiles → accent fallback (consistent with the donut, whose absent-color slices all fall to one muted gray — the platform has NO per-tile palette, by design).

**Class / how to apply:** (a) a KPI/spec that references a ctx dim must have that dim PRESENT in the mode where the spec is shown — a mode effect that clears a dim + bar-visibility gating can leave a `$ctx` ref resolving to '' (→ 0 for cagr/range). For "over the full series" metrics, drive them from a span-derived dim, not the user's range selectors. (b) The OLD bundle/data is the parity oracle for the INTENT (dynamic span, not a literal). (c) `??` vs `||` matters wherever a value can be an empty STRING from a lookup/display join — `??` keeps '' and silently produces transparent fills / unset pins; prefer `||` for color/string fallbacks. (d) A custom CSS-flex chart with `height:100%` needs every ancestor up to the height-constrained box to carry height — an unstyled a11y/semantic wrapper breaks the chain (apex charts mask this by self-measuring). Probe tell: DOM blocks have size but `backgroundColor: rgba(0,0,0,0)` = empty-color bug; blocks at height 0 = height-chain bug.
