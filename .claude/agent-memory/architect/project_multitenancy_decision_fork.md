The branch `feat/tenant-agnostic-platform` delivers **tenant-AGNOSTIC** (engine carries no Geostat identity; brand=tokens, locale=LocaleString, data=config â†’ a rebrandable single-tenant deploy per agency). It does **NOT** deliver **multi-tenant isolation** (many agencies, one instance, isolated). Verified 2026-06-27: **no `tenant_id` in any of the 35 `ops/postgres/migrations/V*.sql`**, `config.site_config` is flat key/value (no tenant scope), JWT carries roles but **no tenant claim**, token override is global `[data-tenant]` not per-request-resolved.

**Why:** the branch name implies an isolation model the schema cannot enforce. Building features over the ambiguity risks a costly late `tenant_id`-threading retrofit (the most invasive change a data platform can make late).

**How to apply:** treat "single-tenant-per-deploy vs multi-tenant-SaaS" as a P0-DECISION ADR (one-way door) BEFORE more features stack. If SaaS: tenant_id on `config.*`+`stats.*` with RLS (expand-contract), tenant claim in JWT + `AuthContext` port in RenderContext (injected by app, never core â€” Law 3), per-request theme resolution, Constructor tenant context.
