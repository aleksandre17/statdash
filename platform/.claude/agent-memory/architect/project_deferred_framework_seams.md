Seams that exist as published contracts but have NO production consumer yet — deliberately deferred per the project's YAGNI + Protected-Variations discipline. Do NOT treat these as "dead code to wire up" without a real second consumer.

- **`NodeRegistry.getByCapability(cap)` / `getCaps`** — capability routing exists and is tested, but only 2 of ~17 slice metas declare `caps` (`gauge`, `text`). Root cause of its non-use is the unpopulated `caps` data, NOT missing wiring. Activation: populate `caps` across all slices as a deliberate data pass, THEN wire palette grouping.
- **`node:status` event + `useNodeStatus` hook** (`engine/react/src/engine/useNodeStatus.ts`) — the *published* half of the deferred NodeStatusContext. SectionShell's inline ADR (Option D) keeps the section a structural container with no child-data awareness. Activation trigger: the FIRST real aggregate consumer (collapse-section-when-all-panels-empty, or disable-section-export-on-empty). Then panels adopt `useNodeStatus` (Strangler-Fig, one at a time) and a NodeStatusContext aggregator is built.
- **`describeApp()` / `propSchemaToJsonSchema` / `PropSchemaForm`** — schema-driven Constructor-form seam. `PropSchemaForm` (`engine/react/src/components/PropSchemaForm.tsx`) renders a property form from any `PropSchema`. Activation: migrate `apps/panel` hardcoded react-admin forms (e.g. `SectionEdit.tsx`) to consume `nodeRegistry.getSchema(type)` → `PropSchemaForm` (Strangler-Fig).
- **`ctx.ui` / `UIRegistry` (UI-primitive DI)** — open keyed registry BUILT 2026-06-17 (`engine/react/src/engine/types/UIRegistry.ts`). NOW FULLY CONSUMED (no longer a deferred seam): 6 shells call `useUI(ctx.ui, 'key')` — `chart`/`table`/`gauge`/`map`/`kpi-strip` use `'empty-state'`/`'export-bar'`, `georgraph` uses `'panel-layout'`. The `useUI(ui, key)` hook (`engine/react/src/engine/useUI.ts`) wraps `ui.get(key)` in `useMemo` (react-hooks/static-components stability). Mirrors NodeTypeMap→NodeRegistry: empty `UIComponentMap` merging interface (key→PROPS) augmented via `declare module '@geostat/react'`, platform defaults `'panel-layout'|'empty-state'|'export-bar'` self-declared in the same file. Runtime `UIComponentRegistry implements UIRegistry` with `get<K>(key)` (fail-fast throw) / `register<K>` / `has(string)`, Map-backed. `createDefaultUI()` (`engine/react/src/engine/createDefaultUI.ts`) returns a populated instance. `UIRegistryOverrides = Partial<Record<keyof UIComponentMap, ComponentType>>`; SiteRenderer applies overrides via `register()` in a useMemo (key-erased setter cast at the one loop boundary — correlated-union TS limitation). Lives in `RenderContext` half **B (runtime services)** — never serializable. `NodePageRenderer`'s `ui` override prop still has zero app callers (the override path is the deferred half; the get-side is live). Do NOT add `getLayout` to NodeRegistry (PanelLayout is not a config-addressable node; SRP). Do NOT add Geostat-specific components to `UIComponentMap` in engine/react (Law 3 agnostic). NEXT EVOLUTION under discussion: replace string keys with `InjectionToken<T>` (Angular/NestJS/tsyringe-style typed tokens, co-located with components) so call sites need no generic — see arch proposals.
- **`SwitchNode`/`ConditionalNode`** — NOT built. No Phase-1 config needs config-time conditional child selection; it would be the first if-in-config mechanism and needs its own ADR (dataShape discriminant vs sandboxed expr; Constructor-serializability). Build only when a real config needs it.

**Why:** These were evaluated in the 2026-06 framework-architecture audit; each is correct-but-speculative until its trigger fires.
**How to apply:** When asked to "use" one of these, first confirm the activation trigger is real; otherwise it stays a published seam, not a consumed one.
