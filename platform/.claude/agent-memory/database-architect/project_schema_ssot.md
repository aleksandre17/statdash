The statdash schema's Single Source of Truth is the Flyway migration set at `national-accounts/ops/postgres/migrations/` — `V1__extensions` … `V25__release_vintage` (head as of 2026-06; V18 unlocked SCD-2, V22 tightened dim_key validate to is_current, V23/V24 expand-contract'd classifier hierarchy from id-chain to parent_code/code_path, V25 added vintage/release). The `apps/api` route code (`apps/api/src/routes/stats/*.ts`, `db.ts`) targets THIS schema exactly: it reads `stats.classifier.id`/`parent_id`/`path`, `stats.observation.dim_key @>` GIN, `time_period_date` partition pruning, `app.pg.query()` via `@fastify/postgres` (node-postgres `pg`).

**V8/V9 additions (2026-06, this agent):** V8 added `stats.observation.obs_attribute jsonb NOT NULL DEFAULT '{}'` (open SDMX attr bag; seqPos now lives here, not dropped) + `stats.observation_revision` append-only audit table with a BEFORE UPDATE trigger `stats.capture_observation_revision()` (no FK — hypertable can't be FK target; references by logical key dataset_code+time_period+dim_key_hash; `revised_by` from `app.revised_by` GUC). V9 did NOT drop/rebuild the generated partition column — V4 ALREADY backs `time_period_date` with `stats.parse_time_period(time_period)` (NOT `::date`), so V9 just `CREATE OR REPLACE`d the parser to add S (semi-annual) + W (ISO week) and widened the `obs_time_period_fmt_chk` CHECK (drop+re-add, superset regex). The partition column was never touched — safest path.

**Why:** A 2026-06 task asked to "formalize the schema into versioned migrations" at `platform/apps/api/migrations/` with a bespoke TS runner and a divergent schema (uuid classifier PK, no dim_key_hash, obs unique on `(dataset_code,time_period,dim_key)`, obs_value double precision, no validation triggers). That would have created a SECOND, conflicting migration system and a second schema definition — a direct SSOT violation. The schema was already formalized.

**How to apply:**
- Before writing ANY migration, confirm whether the schema already exists in `ops/postgres/migrations/`. It does. Treat it as canonical.
- Key canonical facts the routes depend on: classifier `id BIGINT GENERATED ALWAYS AS IDENTITY` (NOT uuid); LTREE `path` trigger-materialized; observation is a TimescaleDB hypertable on `time_period_date` (GENERATED from `time_period` via `stats.parse_time_period`); `dim_key_hash = md5(dim_key::text)` GENERATED STORED; unique index `uq_observation_series (dataset_code, time_period, dim_key_hash, time_period_date)` — partition col MUST be in the unique index on a hypertable; `obs_value NUMERIC`; obs_status A/P/E/R/M + obs_conf F/C/N/D; dim_key validated by `stats.validate_observation_dim_key()` BEFORE trigger against DSD + classifier; SDMX TIME_PERIOD format CHECK.
- The genuinely-missing piece is a per-dataset DISPLAY hint table. Design doc `docs/architecture/future/01-database/obs-schema.md` already specifies the display split as `obs.dim_display (member_id, locale, display jsonb)` — id-keyed + per-locale, NOT `config.display (dataset_code, dim_code, hints)`. Reconcile any new display table against that doc, as a NEW Flyway migration `V6__*`, never a fork.
- Migration tool is Flyway (`flyway_schema_history`), run by `ops/compose/infra/services/flyway.yml`. There is NO bespoke `config.migrations` table or TS runner, and adding one is the fork to avoid.

The geostat bundles (the seed source) are NOT uniform: GDP facts are `{time:number, measure, value, obsStatus:'A'|'P'}` with code-keyed classifiers; regional/accounts use SURROGATE-id classifiers (`Record<id,{code,parent?}>`) and id-keyed display, with facts referencing numeric ids (`geo:number, sector:number`). A seed must handle both the code-keyed and id-keyed bundle shapes, map ids→codes for dim_key, and respect the dim_key validation trigger (every dim_key value must pre-exist in stats.classifier).

**Unit-of-measure SSOT (Gap 3):** per-measure `unit` + precision is structurally defined in `apps/geostat/src/data/metrics.ts` via `registerMetric({ unit:{ka,en}, agg, methodology })` — but that file is side-effectful AND imports `@geostat/engine`, so the seed must NOT import it (engine coupling + ACL violation — see [[feedback-seed-etl-decoupling]]). The chosen pattern: the seed owns a small hand-maintained mirror `apps/api/scripts/seed-units.ts` (`unitFor(code) → {unit_measure, decimals}`) and writes those into `stats.classifier.metadata` of each MEASURE member (Option B: one unit per measure = normalized SSOT home; `obs_attribute` is the per-observation override fallback). The raw.ts bundles carry unit only as Georgian COMMENTS (`მლნ ₾`, `%`, `$`), never structured — do not try to parse them.

**Seed file split (400-line bloat ceiling):** `apps/api/scripts/seed.ts` is a thin orchestrator; cube write helpers (upsertDimension/Classifier/Display/Observation, normalizeStatus, timePeriod, LOCALE) live in `seed-helpers.ts`; unit map in `seed-units.ts`. The post-edit-laws hook HARD-BLOCKS any file >400 lines — keep these split.
