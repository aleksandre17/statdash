### Symptom B — active/selection area lands on the wrong region — VERDICT: REPRODUCED, DETERMINISTIC (root-caused)
**What the user sees.** On the canvas you can select ONLY the leaf items — the 4 KPI cards and the 2 filter controls (year, sector). Clicking the chart, the map, either data table, a section, a grid, the page-header, the filter-bar or the kpi-strip selects **nothing** — the click falls through to bare `.canvas-root`. The whole-node selection "active area" for every one of those elements is collapsed to a 4×4-px dot stacked at viewport **(0,0)**, hidden behind the studio topbar (the "Strata" logo). So the highlight/hit region is not merely offset — it is degenerate and parked in the top-left corner.

**Determinism.** 16 of 16 whole-node frames collapse on every boot/measure. Not racy, not scroll/zoom/transform dependent. (Confirmed the empirical user path too: Playwright click on the "Select chart" affordance resolves to a button at (0,0) and is **intercepted by `<header role="banner">`** → physically unclickable.)

**Geometry fingerprint (the decider).** The offset is NOT scroll-delta-shaped and NOT a constant translation. It is a **zero-measurement collapse**: each affected frame gets `rect = {left:0,top:0,width:0,height:0}`, min-clamped to 4×4 at overlay origin. Per-hitbox inline style is `left:-357px; top:-55.75px; width:0; height:0` (i.e. `elementRect(0,0,0,0) − overlayOrigin(357,55.75)`). This tells the debugger: it is a **box-resolution/zero-size bug, not a coordinate-transform bug** — do NOT chase scroll/zoom math.

**Root cause (mechanism-precise, file:line).**
- The overlay measures each node frame from its renderer anchor: `rootEl.querySelector([data-part-node-id="<id>"])`, then `const box = anchor?.firstElementChild ?? anchor` — `platform/apps/panel/src/canvas/CanvasOverlay.tsx:247-254`. The comment (`:249`) states the intent: *"display:contents anchor has no box — measure its first element child."*
- BUT the renderer emits **TWO nested `data-part-node-id` `display:contents` wrappers per node** (verified: depth-0 and depth-1 both `display:contents`, `getBoundingClientRect()`=0×0; the real boxed content — `.chart-wrap` 536×311, `.kpi-strip` 1091×157, `.panel-col` 538×446 — is at **depth-2**). `firstElementChild` unwraps only ONE contents level → lands on the *second* contents wrapper (still 0×0) → `rel(box)` = 0×0 → frame collapses. The written compensation is **off-by-one-level against a double-wrap**.
- **Why leaf items escape:** the value/sourced item anchor (`data-part-field`/`data-part-index`) is wrapped only **ONCE** (verified: 1 contents wrapper, real box at depth-1), so the identical `firstElementChild` compensation at `CanvasOverlay.tsx:271` reaches the box. Node anchors are double-stamped; item anchors are single-stamped.
- **Origin of the double node-anchor:** `PartAnchor` (`platform/packages/react/src/engine/partAnchor.tsx:82-90`) emits exactly ONE `<div display:contents data-part-node-id>` per call. Two nested identical ones ⇒ the same child node is anchored **twice** — once by the canvas node-anchor middleware and again by the parent slot-part render path (the ADR-041 Phase-4 "anchor merge" — see `CanvasOverlay.tsx:1-21` header + `:239-284` `frameNode`/`walkNodes` dual pass). This is the seam the debugger should confirm (which of middleware vs slot-part is the redundant stamp).
- **Two candidate fixes** (debugger's call): (1) de-dupe so a node is stamped with `PART_NODE_ID_ATTR` exactly once; and/or (2) make `CanvasOverlay.tsx:250`/`:271` box-resolution robust — descend through ALL leading `display:contents` wrappers, not one level (`while (box && getComputedStyle(box).display==='contents') box = box.firstElementChild`). (2) is defense-in-depth and fixes the symptom regardless of wrapper count; (1) removes the grammar duplication root (ADR-041/Law-10 — a node anchored twice is a containment-grammar smell).
- **Regression guard to add** (currently absent — this shipped green): a fitness/e2e asserting every whole-node frame has a non-degenerate rect (width>0 && height>0) matching its rendered content box. `canvasChromeSelectable.fitness.test.tsx` exists but did not catch the node-frame collapse.

**Evidence.** `work/authoring-truth/0109/00-studio-boot-insert-surface.png` (canvas), `B-01-nodeframes-collapsed-at-origin.png` (node frames outlined red = all collapsed top-left / invisible; item frames outlined green = the only selectable targets: 4 KPI cards + 2 filter controls). Console: 0 errors (2 benign React-Router future-flag warnings).

### Symptom A — intermittent freeze ("ზოგჯერ იჭედება") — VERDICT: HARD FREEZE **NOT** REPRODUCED; transient main-thread jank + a duplicate-fetch storm REPRODUCED
**Reproduced (deterministic).** Every page-tab switch triggers a burst of main-thread long tasks: ~430ms cumulative block per settle, individual tasks 100–449ms (measured via `longtask` PerformanceObserver — the reliable signal). Preview-mode toggle (live↔structure): ~170ms/toggle. During these windows the UI is unresponsive (jank), which is the most likely thing the owner experiences as "იჭედება."
**Reproduced (deterministic) — the amplifier: a duplicate-fetch storm.** Every config/classifier/cube-profile/dataset/observation request fires **≥2×** (identical URL, back-to-back — no in-flight dedup; StrictMode double-invoke in dev is part of it but not all). `GET /api/stats/classifiers/measure` fired **~18×** across the walk; each page switch re-fetches ALL three cubes' profiles + datasets + observations even for the page not shown. No 429s seen, all 200/304 — so it is client-side fan-out, not rate-limiting. On the owner's machine (real latency, larger data, longer session) this fan-out can saturate the FetchScheduler budget and stall data-dependent renders → perceived freeze. This is the highest-value Symptom-A lead even though I did not deadlock it.
**NOT reproduced: a hard/unrecoverable freeze.** Tried and the app always recovered, fully responsive (`setTimeout(0)` lateness 0–5ms, 0 errors) after: (a) 6 sequential page switches; (b) **8 page switches stacked with no await** (React coalesced to the final page, settled in 636ms); (c) 6 rapid preview-mode toggles; (d) rapid selection of all 16 node frames in a tight loop.
**Tried-list / what may differ from the owner's state.** NOT covered: real HTML5 **drag-drop** from the palette / node move (Playwright cannot reliably synth native dragstart/dragover — and this path is the one most coupled to Symptom B: dragging re-enables every dropzone, and with all node frames stacked at origin the dropzone geometry there is degenerate → a plausible thrash/freeze locus worth a manual walk). Also NOT covered: long-session memory/listener growth (the overlay attaches a `ResizeObserver` + `window.resize` per mount — `CanvasOverlay.tsx:326-334` — leak-audit on repeated page mounts is warranted); very large bound datasets; production build. My env: automation-driven, short session, fast host — RAF-based wall timings were confounded by background-tab RAF throttling, so I relied on longtask timings only.

**Evidence.** MCP network log (duplicate request pairs 999/1000, 1001/1002 …; `classifiers/measure` ~18×). Longtask stats inline above. No console errors throughout.
